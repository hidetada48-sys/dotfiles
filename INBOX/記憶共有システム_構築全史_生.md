# 記憶共有システム 構築・運用全史（生の記録）

収集日：2026-04-11
対象：basic-memory × Google Drive による PC 間記憶共有システムの構築経緯
収集方針：割愛・要約せず、残存する会話・エラー・感情・判断の瞬間をそのまま時系列に

※ Claude Code はセッション終了後に会話ログが消える。残存するのは「コミット記録」「basic-memory ノート」「INBOX に保存した断片」のみ。消えている部分は「消滅」として明記する。

---

## ▼ そもそもの出発点：なぜ「記憶」が必要だったか

Claude Code はセッションをまたいで会話を覚えていない。毎回「前回どこまでやりましたか？」から始まる。
複数の PC（Linux / Windows）を使っていると、さらに状況がひどくなる。

- Linux で「Xブックマーク収集スクリプトを直した」
- Windows で Claude を起動する
- Claude は何も知らない。処理済みのブックマークを「未処理」として再取得しようとする

この問題を解決するために、段階的に「記憶を外部に持ち出す」仕組みを作り始めた。

## ▼ Phase 1：最初の壁——Windows でパスが合わない（2026-03-04〜03-11）

**2026-03-04**（コミット `8bb93e9`：`スクリプトをWindows環境に対応`）

Linux で作ったスクリプトを Windows で動かしたら動かなかった。

問題はハードコードされたパス：
```bash
# こう書いていた
/home/hidetada48/.x-bookmark-sync/...
```

Windows では `/home/hidetada48/` なんてパスは存在しない。

さらに rclone（Google Drive 同期ツール）のインストール場所も Windows と Linux で違う。Linux は `rclone` でそのまま動くが、Windows は WinGet 経由でインストールすると深いフォルダの中に `rclone.exe` が埋まっている。

修正内容：
```bash
# パスをハードコードせず動的に取得
RCLONE_PATH=$(find "$HOME/AppData/Local/Microsoft/WinGet/Packages" -name "rclone.exe" 2>/dev/null | head -1)
if [ -n "$RCLONE_PATH" ]; then
  export PATH="$PATH:$(dirname "$RCLONE_PATH")"
fi
```

「どの PC・OS でも同じスクリプトが動く」を目指してここから何度も修正が続く。

**2026-03-10**（コミット `80530a3`）

SessionStart / Stop フックを settings.json に登録。
「Claude Code 起動時に自動でダウンロード、終了時に自動でアップロード」の仕組みが動き始める。

## ▼ Phase 2：最初の大きな方針転換——claude-mem を捨てる（2026-03-23）

**2026-03-17（会話ログ：現存）**

この日は Xブックマーク収集スクリプトの改修がメイン。
しかし「processed_ids.json が PC ごとに別々になっている」という問題が発覚した。

具体的に何が起きていたか：
- Linux で「このブックマークは取得済み」と記録
- Windows の Claude は processed_ids.json を知らない
- 同じブックマークを Windows でも取得しようとする → 重複

対処として `gdrive-download.sh` と `gdrive-upload.sh` に processed_ids.json の同期を追加。

これが「ひとつのファイルを Google Drive 経由で同期する」パターンの反復適用であり、後の記憶システム設計につながる発想の原型。

---

**2026-03-23（会話ログ：消滅）**（コミット `300640e`：`basic-memoryによる記憶共有の仕組みを整備`）

会話は消えているが、このセッションで起きた判断は大きい。

`claude-mem`（プラグイン）を廃止し、**basic-memory MCP** に乗り換えた。

なぜ乗り換えたか。basic-memory 導入記録にこうある：

> basic-memory はChromaDBの代替として導入したMCPサーバーです。
> - バックエンド: SQLite + FastEmbed（サーバー不要）
> - セマンティック検索（意味的検索）が使える
> - 暴走しない・安定している
> - 導入日: 2026-03-23

「暴走しない・安定している」という一言に経緯が滲んでいる。それ以前の何かが暴走したり不安定だったのだろう。

basic-memory の実体は SQLite の DB ファイル1つ。これを Google Drive で同期すれば PC 間共有できる。コミット内容：

```
- gdrive-upload/download.sh: basic-memory ノートを同期対象に追加
- gdrive-upload/download.sh: 廃止済みの claude-mem.db 同期を削除
- settings.json: claude-mem プラグインを無効化（false）
```

## ▼ Phase 3：「記憶して」という言葉に反応させる（2026-04-02）

**2026-04-01（会話ログ：現存断片）**

Discord 連携のテスト中に発覚した問題。

ユーザーから「Discordからメッセージを送っても届かないし返信がない」という指摘。

調査したら原因が判明した：

> Claude Code が**能動的に確認しに行かないとメッセージに気づけない**。Discord → Claude Code への「自動通知」がない状態。このセッションで「確認して」と言ったときだけ返信できる。

さらに翌日（4/2）、公式ドキュメントを読んで別の問題が発覚。

```
README Step 6 に明記：
  claude --channels plugin:discord@claude-plugins-official
```

`--channels` フラグなしで起動していた。このフラグがないと Discord メッセージはセッションに届かない。

「なぜ届かないんだろう」と試行錯誤していた時間が、ドキュメントを読む一手で解決した。

**2026-04-02（会話ログ：現存断片）**（コミット `6f2f0d6`）

「記憶して」という言葉を Claude が検知して basic-memory に保存する仕組みを実装した。

```python
# UserPromptSubmit フックで「記憶して」を検知
if '記憶して' in p:
    print('[記憶] mcp__basic-memory__write_note を使ってbasic-memoryに保存してください。')
```

この時点の設計思想：「**保存も呼び出しもフックで制御する**」

保存：UserPromptSubmit フックで「記憶して」を検知 → 保存を促す
呼び出し：別のフックで「記憶を参照して」を検知 → 呼び出しを促す

## ▼ Phase 4：設計の欠陥が露呈・トラブル発覚（2026-04-07）

**2026-04-07（会話ログ：現存）**

この日は Xブックマーク収集を実行しようとして、いきなりエラーに当たった。

```
No module named 'playwright'
```

原因は全然別のところにあった。

```
~/.bashrc に以下が入っていた（2026-03-28 に browser-use をインストールしたときの残滓）：
  export PATH="/home/hidetada48/.browser-use-env/bin:/home/hidetada48/.local/bin:$PATH"

→ python3 コマンドが browser-use 用の仮想環境を参照していた
→ playwright がそこにはインストールされていないのでエラー
```

browser-use はインストールした当日（3/28）しか使われていなかった。それから10日後に PATH が壊れているのに気づいた形。

修正：
```bash
# ~/.bashrc から削除
# export PATH="/home/hidetada48/.browser-use-env/bin:...

# fetch_bookmarks.py のシバン行を変更
# 変更前：#!/usr/bin/env python3   ← PATH 依存
# 変更後：#!/usr/bin/python3       ← 絶対パスで指定
```

---

ブックマーク収集が終わり、ふと「記憶システムがちゃんと動いているか確認しよう」という流れになった。

ここで大きな問題が発覚した。

**Windows PC（miryo）の記憶が全く同期されていない。**

調査すると、Windows の `~/.claude/` の中身が古いままだった：

| ファイル | Windows の実態 |
|---|---|
| `~/.claude/settings.json` | 3/26 版の古い実ファイル（シンボリックリンクではない） |
| `~/.claude/CLAUDE.md` | 一部欠落している古い実ファイル |
| `~/.claude/scripts/` | 3/4 の古いスクリプト群 |

Linux では dotfiles へのシンボリックリンクが正しく設定されていたので、Linux での変更はすべて反映されていた。Windows はそれができていなかった。

結果として起きていたこと：
- `gdrive-download.sh` が古いバージョン → basic-memory のダウンロードが実行されていない
- `UserPromptSubmit` フック（「記憶して」トリガー）が settings.json に存在しない
- CLAUDE.md の記憶参照ルールが欠落している

「ずっと記憶が同期されていると思っていたが、Windows からは全く記憶が使えていなかった」という状態。

**当日の緊急修正**：
- `gdrive-download.sh` / `gdrive-upload.sh` を dotfiles の最新版で手動上書き
- 手動ダウンロードで Linux 側の記憶を初めて Windows に持ち込む

---

**同日、別の問題も発覚**：フックの誤発火

「記憶して」を含む引用テキストが会話に出てくると、フックが誤発火する。

例：「Claudeは『記憶して』という言葉に反応して…」と説明文を書くと、フックが「保存してください」と割り込んでくる。

この問題を受けて設計を見直した。

**変更前の設計**：
- 保存：フック（UserPromptSubmit）で「記憶して」を検知してトリガー
- 呼び出し：フック（キーワード検知）でトリガー

**変更後の設計**：
- 保存：フック（UserPromptSubmit）でトリガー ← これは継続
- 呼び出し：**CLAUDE.md のルールとして明記**（フックを削除）

設計判断の根拠（コミット `29dc248` に残っている）：

> 理由：「行動ルール」は CLAUDE.md の役割。フックはアクションのトリガーに使うべき。

CLAUDE.md に追加されたルール：
```markdown
## 記憶の参照ルール
- 過去の作業・記憶・経緯について聞かれたら、必ず以下の両方を確認してから回答する
  1. auto-memory: ~/.claude/projects/【現在のプロジェクトパス】/memory/MEMORY.md とその参照ファイル
  2. basic-memory: mcp__basic-memory__recent_activity および関連ノートを read_note で読む
- キーワードの有無に関わらず、記憶に関わる質問には必ずこの手順を踏む
```

## ▼ Phase 5：「書いた瞬間に同期」と「自動セットアップ」（2026-04-09〜04-10）

**2026-04-09（会話ログ：現存断片）**

トラブルを受けて「もっと堅牢にしよう」という方向に動く。

### 問題1：記憶を書いても即座に別 PC で使えない

セッション終了時のアップロードだけでは不十分。「記憶して」→ write_note → でも Stop まで Drive にアップロードされない。

対処：PostToolUse フックに `mcp__basic-memory__write_note` のマッチャーを追加。

```json
{
  "matcher": "mcp__basic-memory__write_note",
  "hooks": [{
    "type": "command",
    "command": "bash ~/.claude/scripts/gdrive-upload.sh"
  }]
}
```

書いた瞬間に自動アップロード。

（コミット `1298c95`：`「記憶して」指示時にGoogle Driveへ自動アップロードする設定を追加`）

---

### 問題2：Windows のシンボリックリンクを手動で設定しないといけない

4/7 のトラブルの根本原因はここだった。「設定し忘れ」が起きた。

解決策：SessionStart フックに `setup.sh` を追加。起動するたびに自動でシンボリックリンクを確認・作成する。

```bash
# すでにリンクがあればスキップ、なければ作成
if [ ! -L "$HOME/.claude/settings.json" ]; then
  ln -sf "$DOTFILES/claude/settings.json" "$HOME/.claude/settings.json"
fi
```

設計の考え方（作業記録ノートより）：

> gdrive-download.sh は Google Drive 同期専用。セットアップ処理は分離すべき。
> ツール名をハードコードせず、settings.json から動的に検出する設計。
> Windowsでは初期セットアップ後、次回起動から setup.sh が全自動でセットアップを完了させる。

（コミット `cdee807`：`SessionStart自動セットアップ：setup.sh追加・フックに登録`）

---

**2026-04-10**（コミット `ab11979`：`setup.sh：Windows対応（ジャンクション・ハードリンク）`）

Windows でセットアップを実行して、また新しい壁にぶつかった。

```
ディレクトリのシンボリックリンクが失敗していた
  ~/.claude/hooks → 実ディレクトリとして作成
  ~/.claude/skills/* → 実ディレクトリとして作成
```

Windows では**ディレクトリの** `ln -s` は管理者権限がないと作れない。

調査で「ジャンクション（Junction）」という Windows 独自の方法を見つけた。管理者権限不要でディレクトリリンクを作れる。

```bash
# OS を自動判定して処理を分岐
if [[ "$OSTYPE" == "msys" ]]; then
  cmd /c mklink /J "リンク先" "dotfilesパス"
else
  ln -s "dotfilesパス" "リンク先"
fi
```

## ▼ 現在の構成（2026-04-11 時点）

```
【SessionStart】
  setup.sh          → シンボリックリンクを自動確認・作成（Windows はジャンクション）
  gdrive-download.sh → basic-memory DB を Google Drive からダウンロード

【write_note するたびに（PostToolUse マッチャー）】
  gdrive-upload.sh   → 書いた瞬間に Google Drive へアップロード

【SessionStop】
  gdrive-upload.sh   → セッション終了時にもアップロード（保険）

【「記憶して」と言うと（UserPromptSubmit フック）】
  system-reminder に write_note を促す指示が届く

【記憶を参照するとき（CLAUDE.md ルール）】
  Claude が自律的に auto-memory と basic-memory を両方確認
```

## ▼ 構築を通じて積み上がった「なぜ？」と「そうか」

**なぜ processed_ids.json が重複するのか**（3/17）
→ PC ごとに独立したファイルを持っていたから。Google Drive で同期するしかない。

**なぜ Windows でスクリプトが動かないのか**（3/4〜）
→ Linux のパスをハードコードしていたから。動的取得に変えた。

**なぜ Discord のメッセージが届かないのか**（4/1〜4/2）
→ `--channels` フラグなしで起動していたから。ドキュメントを読んで判明。

**なぜ記憶が Windows から参照できないのか**（4/7）
→ シンボリックリンクが設定されていなかったから。古い実ファイルを参照していた。

**なぜフックが意図せず発火するのか**（4/7）
→ キーワードが引用文の中にも出てくるから。「行動ルール」はフックではなく CLAUDE.md に書くべきだった。

**なぜ書いた記憶が即座に別 PC で使えないのか**（4/9）
→ Stop 時のアップロードだけでは間に合わないから。PostToolUse マッチャーで即時同期に変えた。

**なぜ Windows のシンボリックリンクが作れないのか**（4/10）
→ ディレクトリリンクは管理者権限が必要だから。ジャンクションで回避。
