## 2. NotebookLM × Claude Code（第1週・第2週の両方に登場）

## 3. コンテキスト管理の5原則（第2週・Anthropic社員Thariq氏の解説）

**何か：** 54万ビューされたAnthropicの内部ベストプラクティス。100万トークンでも**30〜40万トークン付近からコンテキスト腐敗（context rot）が発生**するという重要な事実が明かされた。

**5つの判断基準：**

| 状況 | アクション |
|---|---|
| 次のタスクに移る | 新しいセッションを始める |
| アプローチが失敗した | `/rewind` で読み込み直後に戻る |
| セッションが長くなってきた | 余裕のあるうちに `/compact 方向指定付き` |
| 大量の中間出力が出る作業 | サブエージェントに委任 |
| 関連タスクでコンテキストが使える | そのまま続行 |

**特に重要：** `/rewind` が「最も過小評価されている機能」と断言されている。Escキーダブルタップで失敗した試行錯誤のログをコンテキストから丸ごと消せる。

## 4. Routines機能（第2週）

**何か：** Claude Codeのセッションをテンプレート化し、3種類のトリガーで**PCを閉じても**自動実行できる機能。AnthropicのクラウドインフラなのでローカルPC不要。

- **Schedule** — cronのように定期実行（毎時/毎日/毎週）
- **GitHub Events** — PRマージ・Issue作成でエージェント起動
- **API** — HTTP POSTで外部から起動（Zapier等）

**提案：**
- 今の週次ブックマーク収集はcronで動いているが、Routinesに移行すればPC稼働が不要になる
- GitHubトリガーでPRマージ → ドキュメント自動生成も視野に

## 6. 情報収集パイプライン（第1週・@beku_AI）

**何か：** 寝ている間にAIが公式ブログ・X・Reddit等を巡回・フィルタリング・要約してSlack通知する仕組み。1スキル = 20〜30行のMarkdown、1個15分で作れる。

**設計の骨格：**

- フィルタリング基準（例：Enterprise向けは除外、料金変更は最優先）
- 影響度を「高/中/低」で判定させる
- 朝Slackに通知

**提案：**

- これに加えて「公式ブログ・changelogs」を毎日自動監視するスキルを作れば情報収集の網がさらに大きくなる

## 8. Skill作成のベストプラクティス（第1週・@ai_jitan・@MakeAI_CEO）

**核心は `description` の書き方：**

```markdown
# 悪い例
description: Helps with projects.

# 良い例（ユーザーが実際に言いそうな言葉を入れる）
description: Xの投稿文を作成する。
「ポスト書いて」「ツイート作って」「投稿案ほしい」
などのキーワードで自動起動する。
```

- `description` は「Claudeへの求人票」。ユーザーが実際に言いそうな言葉を散りばめる
- 指示本文は命令形で書く（「〜してください」ではなく「〜しろ」）
- `allowed-tools` で権限を最小化するのがセキュリティ的に正しい
