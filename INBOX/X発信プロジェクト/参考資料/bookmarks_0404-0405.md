「AIに仕事を頼む」って言われても、正直ピンとこなかったんですよね。
ChatGPTもClaude Chatも使ってはいます。でもやっていることは結局「質問して、答えをもらって、コピペして、自分で整える」の繰り返し。便利は便利だけど、「仕事を任せた」感覚はありませんでした。
Claude Cowork（以下、Cowork）を触り始めて、その感覚が変わりました。フォルダを指定して「この中の請求書を種類別に分けて、Excelで一覧表も作って」と頼んだら、本当にファイルが仕分けされて、Excelができあがっていました。自分のPC上で。
これは会話ツールじゃない。仕事を渡す相手です。
ちなみにこの記事では、Anthropicの公式ドキュメント「Skill Authoring Best Practices」にある設計思想（スキルは3段階でロードされるから、100個入れてもトークンを圧迫しない、コンテキストウィンドウは公共財、など）まで踏み込んでいます。「Coworkの基本は知ってるよ」という方も、後半まで読んでもらえたら新しい発見があるんじゃないかと思います。
この記事では、Coworkをまだ触ったことがない人が迷わず始められるように、そしてすでに使っている人が「こういう使い方もあるのか」と思えるように、できるだけ具体的にまとめていきます。

**そもそもCoworkって何？ Chat・Code・Coworkの違い**

Claudeには3つの使い方があります。ここを最初に整理しておくと、Coworkの立ち位置がはっきりします。
Claude Chat は会話相手です。プロンプト（指示文）を入れて、答えが返ってきます。ブレスト、下書き、ちょっとした調べものに向いています。ずっと会話の中にいる前提です。
Claude Code は開発者向けのツールです。ターミナル（プログラマーが使う黒い画面）の中で動いて、コードを書いたり、テストを実行したり、GitHub（ソースコードの管理サービス）にプッシュしたりします。技術者じゃないと正直とっつきにくいんですよね。
Claude Cowork はその中間というか、新しい位置づけです。Claude Codeと同じ自律実行エンジン（自分で計画を立てて、自分で手を動かして、完成物を納品する仕組み）が裏で動いているんだけど、インターフェースはデスクトップアプリのGUI（グラフィカルな操作画面）です。技術的な知識がなくても使えます。
Anthropic（Claudeの開発元）の公式説明がわかりやすいです。
Coworkはローカルファイル、フォルダ、日常使うアプリケーションの間を移動し、複数のソースから情報を統合して、ユーザーがステップを調整することなくタスクを完了する
つまり、こういうことです。
Chatは「会話」。Coworkは「タスク委任」。
この区別が腹落ちすると、Coworkの見え方がガラッと変わります。「AIに質問する」じゃなくて「AIに仕事を渡す」になります。

**5分で動かす最初のタスク**

理屈はいいから、まず動かしてみましょう。

ステップ1: モデルを選ぶ
Coworkの画面右側でAIモデルを選べます。
Sonnet 4.6: 基本的にはこれを使います。速くて、安くて、99%の仕事はこなせます
Opus 4.6: 重たい分析や複雑な判断が必要なときだけ。トークン（利用量）の消費が大きいので常用はもったいないです
Haiku 4.5: 軽い仕事向け。速さ重視です
よく言われるのが「アインシュタインに皿洗いはさせるな」。Opusは本当に重要な仕事専用にしておくのが正解です。
ちなみにCoworkではExtended Thinking（拡張思考、Claudeがロジックを組み立てて深く考えるモード）が自動的に有効化されます。Chat画面のように手動でON/OFFする必要はありません。Claudeがタスクの複雑さを判断して、必要なときに勝手に使ってくれます。

ステップ2: サンドボックスフォルダを作る
デスクトップに「claude-workspace」フォルダを1つ作ります。Coworkの操作範囲はこのフォルダの中に限定されます。それ以外のファイルには触れないので安心していいです。

ステップ3: フォルダの権限を付与する
Cowork内で「Work in a folder」をクリックして、さっき作ったフォルダを選びます。「Allow once（今回だけ許可）」か「Always allow（常に許可）」を選んで完了です。

ステップ4: 最初のタスクを実行する
散らかったダウンロードフォルダの中身を10〜20個、サンドボックスにコピーしてきて、こう頼んでみてください。
このフォルダの中のファイルを種類別にサブフォルダに分けて、ファイル一覧のExcelも作って。
Coworkは画面右側に計画を表示して、自律的に動き始めます。複数のサブエージェント（並列で動く分身）を立ち上げて、同時に処理を進めることもあります。コーヒーを淹れて戻ってきたら、ファイルが整理されてExcelができています。
これが「仕事を渡す」という感覚です。

**毎回ゼロから説明するのをやめる**

Coworkを使い始めてしばらくすると、あることに気づきます。
「また自分の会社のこと説明してる…」
Coworkは新しいセッション（会話窓）を開くたびにまっさらな状態から始まります。先週やった仕事のことも、自分の会社のことも、好みの文体も何も覚えていません。これが「使い捨てチャット」状態です。
この「毎回ゼロからやり直し」感、めちゃくちゃストレスなんですよね。ここを突破するのが、Coworkを使いこなす最初の分岐点だと思います。3つの仕組みを使います。

プロジェクト機能：文脈が蓄積される「箱」
プロジェクトは単なるフォルダじゃありません。ファイル、カスタム指示、スキル（後述）、そして蓄積された文脈がまとまったコンテナです。
プロジェクトなしだと、Claudeは自分のことを何も知りません。毎回ゼロから説明します。永遠に。
プロジェクトありだと、文脈が積み上がります。最初の1週間は丁寧にプロンプトを書いているけど、6週間目には「いつもの感じで」と言えば通じるようになります。
大事なのは、テーマ別にプロジェクトを分けることです。YouTube運用と経理を同じプロジェクトでやると、YouTubeの文体ルールが経理タスクに混入してClaudeが混乱します。これは分けます。

.mdファイルでClaudeに「ビジネス脳」を与える
プロジェクトが箱なら、中身の「性格」はマークダウンファイル（.md、テキストファイルの一種）で定義します。プロジェクトのコンテキストフォルダ（文脈を置く場所）に入れておくと、Claudeが毎回のセッション開始時に読み込んでくれます。
具体的にはこの3つです。
about_me.md：自分は誰で、何の会社をやっていて、顧客は誰で、どうやって収益を上げていて、今の優先事項は何か。Claudeはこれを毎回読みます。毎回です。

例：
```
## 基本情報
- 名前:aaa
- 役割: プロダクト開発
- 主な業務: MEO（Googleマップ検索の最適化）代行事業の立ち上げ

## 顧客
- ローカルビジネス（飲食店、美容室、クリニックなど）
- ITリテラシーは高くないが、集客への関心は強い

## 今の優先事項
- MEOツール群の自動化率を上げる
- note/Xでの情報発信を習慣化する
```

brand_voice.md：どんな文体で書くか、好きなトーン、嫌いな表現、実際の文章サンプル。これがないと「AIが書いた感」丸出しの文章が出てきます。

例：
```
## トーン
- カジュアルだけど中身は濃い。友人に説明するような口調
- 一人称は「僕」
- 「めちゃくちゃ」「ぶっちゃけ」「やってみた」OK

## 避ける表現
- 過度に丁寧な敬語（「させていただきます」）
- 抽象的なビジネス用語だけの文（「DX推進」「シナジー」）
- 上から目線の断定（「〜すべきです」）
```

working_preferences.md：タスクの進め方、ファイルの保存先、出力フォーマットの好み。
これは全部自分で書く必要はありません。Claudeに「一連の質問をして、僕の回答からビジネス脳ファイルを作って」と頼めば、15分のインタビューで完成します。15分の投資で、毎週何時間も節約できます。

設定の3階層を使い分ける
Coworkには指示を入れられる場所が3つあります。スコープ（適用範囲）が違うので、正しく使い分けると一貫した動作になります。
レベル1: Claudeパーソナライゼーション（設定 → 名前をクリック） Chat、Code、Coworkすべてに適用されます。「太字を使いすぎない」「曖昧な表現を避ける」「一次情報を優先する」みたいな、自分の普遍的なクセをここに書きます。
レベル2: Coworkグローバル設定（設定 → Cowork設定 → グローバル指示） Cowork内の全タスクに適用されます。日付フォーマット、ファイル命名規則（アンダースコア区切りの説明的な名前にする、等）、ビジネスに関する質問はサンドボックスフォルダを先に確認する、といったルールをここに。
レベル3: プロジェクト固有の設定 特定のプロジェクトだけに効くルールです。YouTube運用プロジェクトなら「動画イントロには特定のスライドスキルを使う」「進捗は特定のスプレッドシートで管理する」みたいな指示。これが他のプロジェクトに漏れることはありません。

**外部ツールと繋いで実用度を一気に上げる**

正直、ファイル整理だけでも十分便利なんですよね。でも、Coworkの真価は日常使っているツールと繋がったときに出ます。

MCP（Model Context Protocol）コネクタ
MCP（モデルコンテキストプロトコル）は、Claudeが外部アプリに直接アクセスするための仕組みです。コピペでもなく、スクリーンショットでもなく、認証済みの直接接続です。
接続方法はシンプルで、チャット内のプラスボタン → コネクタタブ → 使いたいツールを検索 → ブラウザで認証フロー。これだけです。
対応しているネイティブコネクタの一例：
Slack（チャットツール）
Notion（ドキュメント・データベース管理）
Google Drive（ファイル共有）
Gmail
GitHub（ソースコード管理）
Figma（デザインツール）
HubSpot（CRM、顧客管理ツール）
DocuSign（電子署名）
接続すると、アクションごとに「常に許可 / 承認が必要 / ブロック」を細かく設定できます。何を触らせて、何を触らせないかは自分でコントロールできます。
繋がるとどうなるかというと、例えばこういう使い方ができます。
Slackの#generalチャンネルの今週の未読をまとめて、重要なものだけピックアップして。
Notionの議事録データベースから先週の会議メモを取ってきて、次のアクションアイテムを抽出して。
Google Driveの「提案書」フォルダにある最新のファイルを読んで、要約して。
1つの画面で、複数のツールをまたいだ仕事が完結します。これがCoworkの本領だと思います。

日本の業務ツールとの連携: freee
freee（クラウド会計ソフト）のMCPコネクタが使えるのは、日本のユーザーにとってかなり大きいです。会計・人事労務・請求書データに直接アクセスできます。
例えば「今月の経費を集計してレポートにまとめて」と頼めば、freeeのデータを引っ張ってきて、整理して、レポートファイルを生成します。手作業で管理画面とExcelを行き来する時間がごっそり消えます。

Zapier MCP：「うちのツール対応してないじゃん」への回答
ネイティブコネクタにないサービスと繋ぎたいとき、Zapier MCP（ザピア）が橋渡し役になります。Zapierは8,000以上のアプリと連携できるサービスで、そのMCPをCoworkに接続できます。
設定は、Zapierのプラットフォームでクライアントに「Claude Cowork」を選んでMCPサーバーを作成 → 許可するツールとアクションを設定 → 生成されたURLをCoworkのコネクタに貼ります。10分くらいで終わります。
ただ、正直に言うと「8,000アプリ！」という数字ほどのインパクトが日常的にあるかというと、そうでもありません。ネイティブコネクタがあるサービス（Slack、Notionなど）はネイティブのほうが操作の粒度が細かくて安定しています。Zapier MCPの本質は「ネイティブにないサービスとの補完手段」です。
kintone（サイボウズの業務アプリ）やAirtable（オンラインデータベース）のように、日本企業でよく使われているけどMCPコネクタが未対応のツール。こういうものと繋ぎたいときにZapier MCPが効きます。
ただしkintoneのようにアプリ構造が複雑なサービスだと、Zapier経由では操作がもたつくことがあります。単純なレコード追加やステータス更新なら問題ありませんけど、サブテーブルやプロセス管理が絡む操作はZapier側のアクション設計次第になります。この辺は過度な期待はしないほうがいいです。

**プラグインで「専門AI」を手に入れる**

ここからCoworkの使い方がもう一段変わります。
プラグインは、インストールするだけでCoworkに専門能力が追加される仕組みです。「経理のプロ」「営業のプロ」「マーケティングのプロ」がワンクリックで手に入る感覚です。
具体的にどんなことができるのか、部門別にまとめてみます。

Finance（経理・財務）：
/journal-entry：仕訳を作成。借方・貸方と根拠書類をセットで出してくれます
/income-statement：損益計算書を前期比較・差異分析付きで生成します
/sox-testing：SOX（内部統制）監査のサンプル選定とテスト調書を作成します
月次決算管理、勘定科目の照合、経費の差異分析にも対応します

Sales（営業）：
/call-prep：商談前の準備。取引先情報、参加者のリサーチ、アジェンダ案を自動生成します
/draft-outreach：見込み客を調べてから、パーソナライズしたアウトリーチ文を作成します
/pipeline-review：パイプライン（商談の進捗管理）を分析、リスクのある案件をフラグ立てします
/forecast：加重平均の売上予測をベスト・ミドル・ワーストの3シナリオで生成します

Marketing（マーケティング）：
/draft-content：ブログ、SNS投稿、メールニュースレター、LP（ランディングページ）、プレスリリースを作成します
/campaign-plan：キャンペーンの目的、チャネル戦略、コンテンツカレンダー、KPI（目標達成度を測る指標）を一括設計します
/seo-audit：キーワード調査、ページ内分析、コンテンツギャップ、技術チェックをまとめて実行します
/competitive-brief：競合のポジショニングとメッセージングを比較分析します

Customer Support（カスタマーサポート）：
/triage：問い合わせを分類して優先度（P1〜P4）をつけ、担当チームへの振り分けを提案します
/draft-response：状況と緊急度に合わせた顧客向け返信文を作成します
/kb-article：解決済みの問題からナレッジベース（社内の知識集約場所）の記事を自動生成します
/escalate：エンジニアや経営層への報告資料をパッケージングします

Product Management（プロダクト管理）：
/write-spec：PRD（製品要求仕様書）を課題定義、ユーザーストーリー、受入条件付きで作成します
/roadmap-update：RICE、MoSCoW等のフレームワーク（優先順位づけの枠組み）でロードマップを整理します
/synthesize-research：ユーザーインタビューやアンケート結果から気づきを構造化します
/stakeholder-update：経営層向け、エンジニア向け、顧客向けなど相手に合わせた報告書を作成します

ChatGPTにも「GPTs」というカスタマイズ機能があります。でも、CoworkのプラグインとGPTsの違いは決定的です。GPTsは「プロンプトのテンプレート」に近いです。Coworkのプラグインは、自分のPCのファイルにアクセスできて、SlackやNotionと実際に繋がって、コードを実行できます。「答えを教えてくれるAI」と「仕事を実行するAI」の差です。

**スキル：繰り返す作業を「1コマンド」に変える**

「週に1回以上やる作業はスキルにしろ」
毎週同じことを手作業でやってる自分に気づいたこと、ありませんか。これはシンプルだけど強力なルールだと思います。
スキルとは何か。再利用可能なワークフローを1つのコマンドにまとめたもの。裏側は.mdファイル（指示書）とスクリプト（自動処理プログラム）の組み合わせで、Claudeが必要に応じて読み込みます。

スキルの作り方
複雑そうに聞こえるかもしれませんけど、流れは意外とシンプルです。
ステップ1: まず普通にClaudeとタスクをやる。 例えば、毎週やっている「YouTubeの動画URLからトランスクリプト（文字起こし）を取って、Notionにまとめて、X投稿用の要約を3パターン作る」みたいな仕事を、普通の会話でやります。
ステップ2: うまくいったパターンを特定する。 「毎回同じような指示を出していて、毎回うまくいっている」部分を見つけます。
ステップ3: Claudeにスキル化を頼む。 「今のやり方をスキルにして」と言えば、Claudeが.mdファイルを生成します。コードを自分で書く必要はありません。
ステップ4: テストする。 Coworkの内蔵評価機能がスキルあり/なしの出力を比較してくれます。「ベージュの背景を使い忘れた」「フォントの太さが違う」みたいなフィードバックを返して調整します。
ステップ5: 保存して運用開始。 以後はスラッシュコマンド（/スキル名）で呼び出すだけです。

Anthropicが教えるスキル設計の3原則
ここからは、Anthropicの公式ドキュメント「Skill Authoring Best Practices（スキル作成のベストプラクティス）」から、特に重要な設計思想を紹介します。スキルを1つ2つ作ったことがある人ほど、ここの考え方が効いてくると思います。

原則1: 簡潔さが命：コンテキストウィンドウは公共財
コンテキストウィンドウ（Claudeが一度に処理できる情報量の枠）は有限で、スキルの指示、会話の履歴、システムプロンプト、他のスキルのメタデータ（名前と説明文）がすべて同じ枠の中に入ります。つまり、スキルの中に無駄な説明を詰め込むと、他の情報が入るスペースが減ります。
公式の表現がいいです。「デフォルトの前提: Claudeはすでに十分賢い」
スキルに書く情報の1つ1つに対して、こう問いかけます：
「Claudeはこの説明を本当に必要としているか？」
「Claudeがすでに知っていることを繰り返していないか？」
「この段落はトークンコスト（消費量）に見合っているか？」

悪い例（150トークン消費）：
PDF（Portable Document Format）は一般的なファイル形式で、テキスト、画像、その他のコンテンツを含みます。PDFからテキストを抽出するにはライブラリが必要です。利用可能なライブラリはたくさんありますが、pdfplumberは使いやすく…

良い例（50トークン消費）：
```
pdfplumberでテキスト抽出：import pdfplumber
with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
Claudeはそもそも「PDFが何か」も「ライブラリが何か」も知っています。教える必要はありません。

原則2: 自由度を意図的に設計する：橋のメタファー
Claudeをロボットに見立てて、道の安全性に応じてガイドの粒度を変えます。これが公式の「橋のメタファー」です。
崖の横の細い橋（失敗したらダメージが大きいタスク）： 安全な道は1つしかありません。コマンドを一字一句指定して、余計なことをさせません。
例：データベースのマイグレーション（構造変更）
```
このスクリプトをそのまま実行：
python scripts/migrate.py --verify --backup
コマンドを変更したりフラグを追加しないこと。
```
何もない開けた草原（失敗してもリカバリーできるタスク）： 多くの道が正解に通じます。方向性だけ示して、Claudeに最適ルートを任せます。
例：コードレビューのプロセス
```
1. コードの構造と整理を分析
2. 潜在的なバグやエッジケースをチェック
3. 可読性と保守性の改善を提案
4. プロジェクト規約への準拠を確認
```
ほとんどの業務タスクはこの中間のどこかに位置します。判断基準は「このタスクが失敗したときのダメージの大きさ」です。ダメージが大きいほど自由度を下げて、小さいほど上げます。

原則3: 段階的開示（Progressive Disclosure）：必要なものだけ、必要なときに
これがスキル設計で最も知っておくべき仕組みだと思います。
Coworkのスキルは3段階でロードされます。
Level 1: メタデータ（常にロード） スキルの名前と説明文だけです。1つあたり約100トークン。100個のスキルをインストールしても、この段階では10,000トークンしか消費しません。Claudeは「このスキルが存在する」「どういうときに使う」だけを知っている状態です。
Level 2: 指示書（スキルが発動したとき） SKILL.md（スキルの本体ファイル）の中身が読み込まれます。公式推奨は5,000トークン以下、500行以内です。
Level 3: リソースとコード（必要なときだけ） 参照ファイル、スクリプト、データ。Claudeが「これが必要だ」と判断したときだけ読みに行きます。使わないファイルはコンテキストを一切消費しません。

```
pdf-skill/
├── SKILL.md          ← Level 2: 発動時に読み込み（目次のような役割）
├── FORMS.md          ← Level 3: フォーム操作が必要なときだけ
├── REFERENCE.md      ← Level 3: API情報が必要なときだけ
└── scripts/
    └── fill_form.py  ← Level 3: 実行されるが中身はコンテキストに入らない
```
つまりSKILL.mdは「目次」として書きます。詳細は別ファイルに分割して、Claudeが必要なファイルだけ読みにいく設計にします。
ここのポイントは、スクリプト（.pyファイルなど）は実行されるけど、コードの中身自体はコンテキストウィンドウに入らないことです。出力結果だけが返ってきます。だから同じ処理を会話の中でClaudeにコードを書かせて実行するよりも、スクリプト化しておくほうがトークン効率がずっといいです。

スキルを育てる方法
公式が推奨しているのが「Claude A（設計者）とClaude B（テスター）」パターンです。
Claude A（ある会話ウィンドウ）と一緒にスキルを設計・作成する
Claude B（別の会話ウィンドウ、スキルをロード済み）に実際のタスクをやらせる
Claude Bの挙動を観察します。スキルの情報を見つけられているか？ルールを正しく適用しているか？
問題があればClaude Aに戻って「Bがこういう場面で失敗した。どう直す？」と相談します
修正して、また Claude B でテスト
実際の使用で見つかった問題をもとに改善するので、机上の想定よりずっと精度が上がります。

**コネクタを組み合わせて「業務パイプライン」を作る**

1つのMCPコネクタを繋ぐだけでも便利だけど、複数を組み合わせたときに起きるのは「量の変化」じゃなくて「質の変化」です。
具体例を出します。

例: 週次営業レポートの自動化
手でやる場合：
Slackの#salesチャンネルを遡って、今週の商談メモを拾う（15分）
Notionの商談データベースにメモを転記する（10分）
Wordで週次レポートのテンプレートを開いて書き込む（20分）
Google Driveにアップロードする（5分）

Coworkでやる場合：
先週やったのと同じ形式で、今週の営業レポートを作って。Slackの#salesチャンネルから今週の商談メモを取ってきて、Notionの商談データベースに追加して、Docxで週次レポートにまとめてGoogle Driveに保存して。
1行です。プロジェクトに文脈が蓄積されているので、フォーマットも命名規則も前回の踏襲になります。

CLAUDE.mdでパイプラインの品質基準を固定する
プロジェクトフォルダの直下にCLAUDE.mdというファイルを置くと、セッション開始時にClaudeが自動で読み込みます。ここにワークフローの手順書を書いておくと、誰が（どのセッションで）実行しても同じ品質基準で動きます。
新人に引き継ぎ書を渡すような感覚に近いです。「レポートのファイル名はYYYY-MM-DD_weekly_sales_report.docxにする」「売上数値は千円単位で丸める」「レポートの冒頭には先週のハイライトを3行で書く」みたいなルールを書いておけば、Claudeが毎回それに従います。
CLAUDE.mdの設計で意識すると良いのは：
プロジェクト全体のコーディング規約・品質基準はここに書く
特定タスクの詳細手順は別のスキルファイルに分離する（CLAUDE.mdが長くなりすぎると毎回の読み込みコストが増える）
「こうしてはいけない」というアンチパターンも明記しておく（Claudeは「やってね」より「やるな」のほうを忠実に守る傾向があります）

**スケジュールタスクとディスパッチ：自分がいなくても回る仕組み**

スケジュールタスク
「毎朝9時にやっておいて」を設定する方法です。
設定方法：
左サイドバーの「Scheduled（スケジュール）」タブに移動
「New Task」をクリック
タスク名と説明を入力
プロンプト欄にやらせたいことを書く（例：「メールとカレンダーをチェックして、今日のデイリーブリーフを作って」）
実行頻度を設定：毎時、毎日、毎週、特定曜日の特定時刻
対象プロジェクトフォルダを選んで保存
注意点があります。スケジュールタスクは、PCが起動していてCoworkアプリが開いている場合にのみ実行されます。ノートPCが閉じていたら、9時のタスクはPCを開くまで待機状態になります。電源設定でスリープを防ぐ設定にしておくのを忘れないでください。

ディスパッチモード：スマホから自宅PCを遠隔操作
外出先にいるけど、自宅PCのファイルで処理したいタスクがあります。そういうときにディスパッチモードを使います。
スマホのClaudeアプリからメッセージを送ると、自宅のPC上でCoworkがタスクを実行します。会話スレッドが共有されているので、スマホとPCの間でコンテキスト（文脈）が途切れません。
設定：
デスクトップのCowork設定でディスパッチを有効化
ディスパッチメニューの「Keep Awake（スリープ防止）」をON。これを忘れるとPCがスリープしてファイルアクセスできなくなります
ブラウザ操作やコンピュータ操作を許可する場合は個別に設定
使用例： スマホから「レシートフォルダをスキャンしてダッシュボード作って」と送ります。自宅PCのCoworkが処理を実行して、プッシュ通知で完了を知らせてくれます。

**Claude in Chrome：ブラウザ操作という奥の手**

MCPコネクタがないWebサービスを操作したいとき、Chrome拡張（ブラウザの追加機能）が使えます。

セットアップ
Google Chromeのウェブストアで「Claude」を検索 → インストール → ツールバーにピン留め → Coworkの設定で「Claude in Chrome」を有効化します。

できること
Chromeのタブを開いてWebページを自律的に読む
ページの構造（見出し、強調テキスト、レイアウトの階層）を理解する
リンクをクリックして、ページを遷移する
フォームに入力する
競合サイトの定点観測、フォーム入力の自動化、Web上のデータ収集
YouTubeを開いて、おすすめ動画の視聴回数、高評価率、コメントの傾向を取ってきて。
こういう指示もできます。

セキュリティの注意
ここは強調しておきます。Claudeは自分のChromeを使います。つまり、ログイン済みのアカウントにアクセスできます。もし航空券予約サイトで決済情報が保存されていたら、理論上は購入を完了できてしまいます。
対策：
Coworkの設定でブロックリスト（操作を禁止するサイト）を先に設定する
銀行、決済サービス、重要なアカウントのサイトはブロックリストに入れる
Chrome操作中はアクティビティを監視する。新人社員にアカウントのアクセス権を渡したときと同じ心構えで

**Computer Use：画面を見てマウスを動かす**

Claude in Chromeはブラウザ内の操作に限定されますけど、Computer Use（コンピュータ操作）はもう一歩先を行きます。Claudeが画面のスクリーンショットを撮って、マウスを動かして、キーボードを叩いて、デスクトップアプリを直接操作する機能です。
設定 → 一般 → 「Computer Use」をONにします。ただし、ONにする前にブロックリスト（操作を禁止するアプリ）を必ず設定してください。
できることの例：
デスクトップ上の動画ファイルを見つけて、CapCut（動画編集アプリ）にドラッグ&ドロップする
Finderでファイルを検索して、特定のアプリで開く
デスクトップアプリのUIを視覚的に認識して、ボタンをクリックする
使い分けの整理としては、MCPコネクタで対応しているサービスならコネクタを使います。コネクタがなくてブラウザでアクセスできるならClaude in Chrome。それでもダメなデスクトップアプリの操作がComputer Use。この3段階で考えるといいです。
Pro/Maxプランで利用可能で、macOSとWindowsの両方に対応しています。

**フィードバックループと検証：出力品質を安定させる設計パターン**

スキルを作って、コネクタを繋いで、パイプラインを組みます。ここまでくると、次に気になるのは「出力の品質がブレる」問題だと思います。
Anthropicの公式ベストプラクティスが繰り返し強調しているのは「一発で完璧を求めず、ループを回す」ことです。具体的なパターンが3つあります。

パターン1: 実行→検証→修正→再実行ループ
スキルの中に検証ステップを組み込みます。Claudeにやらせっぱなしにするんじゃなくて、やったあとに自分でチェックさせます。
例えばWordドキュメントの編集スキルなら：
```
1. document.xmlを編集する
2. 即座にバリデーション（検証）を実行: python scripts/validate.py unpacked_dir/
3. バリデーションに失敗したら:
   - エラーメッセージを読む
   - XMLの問題箇所を修正する
   - 再度バリデーションを実行する
4. バリデーションを通過したときだけ次へ進む
5. ドキュメントを再構築: python scripts/pack.py unpacked_dir/ output.docx
```
この「実行→検証→修正→再実行」を自動でループさせます。手動チェックなしでも品質が安定する仕組みです。
公式が補足しているのは、バリデーションスクリプトのエラーメッセージは具体的に書くことです。「バリデーション失敗」だけだとClaudeは何を直せばいいかわかりません。「フィールド 'signature_date' が見つかりません。利用可能なフィールド: customer_name, order_total, signature_date_signed」のように書くと、Claudeが自力で修正できます。

パターン2: チェックリスト駆動ワークフロー
複雑なタスクをステップに分解して、Claudeに進捗を追跡させます。
```
リサーチ進捗:
- [ ] ステップ1: ソース文書をすべて読む
- [ ] ステップ2: 主要テーマを特定する
- [ ] ステップ3: 主張をソース間で相互参照する
- [ ] ステップ4: 構造化されたサマリを作成する
- [ ] ステップ5: 引用を検証する（不備があればステップ3に戻る）
```
このチェックリストをスキルの中に書いておくと、Claudeが自分でチェックを入れながら進みます。ステップを飛ばすことがなくなるし、ステップ5で引用に不備があればステップ3に戻るような「分岐」も仕込めます。
コードを含まないタスク（リサーチ、文書作成、レビュー）でもこのパターンは使えます。スクリプトの代わりにスタイルガイド（文体の基準書）をバリデーターとして使い、Claudeに「チェックリストの項目をスタイルガイドと照合して」と指示します。

パターン3: 検証可能な中間出力
いきなり最終成果物を作るのではなく、まず「計画ファイル」を作らせます。
例えば50件のPDFフォームに一括入力するタスクがあるとします。いきなり全部入力させると、フィールド名の間違いや値の矛盾に気づくのが最後になります。
代わりに：
フォームの構造を分析させる（analyze_form.py）
入力計画をJSONファイルとして出力させる（changes.json）
計画ファイルをバリデーションスクリプトで検証する（validate_fields.py）
問題なければ実行（fill_form.py）
完了後に結果を確認（verify_output.py）
「分析→計画作成→計画検証→実行→確認」の5ステップです。ステップ3で問題が見つかればステップ2に戻ります。
この「計画を先に作って、実行前に検証する」パターンは、バッチ処理（大量の一括処理）、破壊的な変更（元に戻しにくい操作）、高リスクな作業（ミスの影響が大きい操作）で特に有効です。

**トークンを無駄にしない実践知**

「なんかもう今週の利用上限に達しちゃった…」という経験、ありませんか。Coworkに送るすべての言葉と、Claudeがファイルから読み込むすべての情報にトークンがかかります。管理が甘いと、週の半ばで上限に到達します。
避けるべき3つのパターンがあります。

ミス1: ベースロードを無視する
自分が1文字もタイプしていない状態で、すでにコンテキストウィンドウの一部が埋まっています。システム指示、有効なツール、接続中のMCPコネクタです。全部カウントされています。コネクタをたくさんONにしていると、それだけで基礎消費が増えます。今のタスクに必要なコネクタだけONにしてください。

ミス2: コンテキスト腐敗
同じチャットタブで何時間も作業を続けると、Claudeは毎回その会話の全履歴を再読み込みします。メールの下書きを終えたあと、同じウィンドウでドバイ旅行の計画を始めたら、メールのコンテキストが旅行の処理に毎回ロードされます。純粋な無駄です。
30〜45分ルール：セッションは30〜45分、1トピックに限定します。話題が変わったら新しいウィンドウを開きます。新しいコンテキスト、新しい予算。

ミス3: 会話で大量処理
100件の請求書をClaudeに1件ずつ会話の中で読ませて処理させると、セッションのトークンがみるみる減ります。代わりに、スキル化してスクリプトで回します。スクリプトの実行結果だけがコンテキストに入るので、トークン消費は桁違いに少ないです。

並列サブエージェントの活用
大きなタスクでは「並列で処理して」と指示します。Coworkが複数のサブエージェント（分身）を立ち上げて、それぞれ別のコンテキストウィンドウで同時に作業します。1つのウィンドウにすべてを詰め込むよりもトークン効率がいいし、処理も速いです。

**外部スキルのセキュリティ：2分でできる安全確認**

GitHubなどからコミュニティが作ったスキルをダウンロードして使えます。これは便利だけど、リスクがあります。
スキルは「Claudeへの指示書」です。もし悪意のある指示が.mdファイルに埋め込まれていたら、プロンプトインジェクション（AIへの不正な指示注入）になります。ファイルの削除、データの外部送信、アクセス権限の昇格。理論上は全部できてしまいます。
Anthropicの公式も「信頼できるソース（自分で作ったもの、またはAnthropicから入手したもの）のスキルのみを使用することを強く推奨」と明記しています。
もし外部スキルを使うなら、この手順を毎回やってください。
スキルの.mdファイルの全文をClaude Chatに貼って、こう聞きます：
このスキルに有害な指示、タスクの範囲外の動作を指示する内容、または不審なパターン（予期しないネットワーク呼び出し、ファイルアクセスパターン、本来の目的と合わない操作）はないか？
2分です。毎回やります。例外なし。
特に注意すべきは、外部URLからデータを取得するスキルです。取得先のコンテンツが変更されれば、もともと安全だったスキルが後から危険になりうります。

**まとめ：今日やる3つのこと**

ここまで読んで「全部やるのは大変だな」と思ったかもしれません。でも、最初の3つだけやれば、Coworkの世界は十分に変わります。
1. フォルダを1つ作って、1つタスクを任せてみる。 デスクトップに「claude-workspace」フォルダを作って、散らかったファイルを入れて、「種類別に整理して」と頼みます。5分で終わります。
2. about_me.mdを作ってClaudeに自分を覚えさせる。 「僕のことをインタビューして、ビジネス脳ファイルを作って」と頼むだけです。15分で、毎回のゼロからの説明がなくなります。
3. 普段使っているツールのコネクタを1つ繋いでみる。 SlackでもNotionでもGoogle Driveでもいいです。1つ繋がるだけで「Coworkってファイル整理だけじゃないんだ」とわかります。
この3つを起点にして、慣れてきたらスキル化→スケジュールタスク→パイプライン構築へ進みます。Coworkの恩恵は使うほど加速します。最初の1歩が一番ハードルが高くて、そこを超えたらあとは勝手に「これもCoworkでやれるな」が見えてくるようになります。

The hook transparently rewrites Bash commands (e.g., git status -> rtk git status) before execution. Claude never sees the rewrite, it just gets compressed output.

Important: the hook only runs on Bash tool calls. Claude Code built-in tools like Read, Grep, and Glob do not pass through the Bash hook, so they are not auto-rewritten. To get RTK's compact output for those workflows, use shell commands (cat/head/tail, rg/grep, find) or call rtk read, rtk grep, or rtk find directly.

**How It Works**
```
  Without rtk:                                    With rtk:

  Claude  --git status-->  shell  -->  git         Claude  --git status-->  RTK  -->  git
    ^                                   |            ^                      |          |
    |        ~2,000 tokens (raw)        |            |   ~200 tokens        | filter   |
    +-----------------------------------+            +------- (filtered) ---+----------+
```

Four strategies applied per command type:
Smart Filtering - Removes noise (comments, whitespace, boilerplate)
Grouping - Aggregates similar items (files by directory, errors by type)
Truncation - Keeps relevant context, cuts redundancy
Deduplication - Collapses repeated log lines with counts
