### Claude Codeを「なんとなく使ってる」から卒業する。Best Practiceを全部まとめた

- 投稿者：@daifukujinji（引用元：@nobel_824）
- 日時：2026-04-03 00:59:55
- URL：https://x.com/daifukujinji/status/2039870478912004430

**ツイート本文（引用元ツイート含む）：**

（引用ツイート：https://x.com/nobel_824/status/2039840802504077655）

Claude Code、使ってますか？
僕の周りでも使っている人は増えてきましたが、まだまだ活用しきれている人は少ないように思います。
かくいう僕も最初はそうでした。とりあえず指示を出して、返ってきたコードをコピペして、うまくいかなかったらまた聞いて…の繰り返し。正直、ChatGPTとの違いがよくわかりませんでした。
でも、GitHub（ソースコードの共有サイト）で公開されている「claude-code-best-practice」というリポジトリ（プロジェクトの保管場所）を見つけてから、使い方がガラッと変わりました。スター数は3万超え。海外の開発者がClaude Codeのベストプラクティスをまとめてくれています。（元はClaude Codeの開発者boris chemyの使い方をまとめたもの）

この記事では、そのリポジトリに掲載されている全87個のTipsを「Claude Codeをこれから使う人・使い始めたばかりの人」向けに、僕なりに噛み砕いて紹介していきます。
ちなみに、このベストプラクティス集は自分のClaude Codeに簡単に取り込めます。一番簡単なのは、Claude Codeを開いてこう打つだけです。

```
https://github.com/shanraisshan/claude-code-best-practice を読んで、
うちのプロジェクトに合うベストプラクティスをCLAUDE.mdに反映して
```

Claude Codeがリポジトリの内容を読み取って、プロジェクトに合った形でCLAUDE.md（後述する「記憶ファイル」）を作ってくれます。コマンドを覚える必要もありません。

**まず知っておきたい：Claude Codeの「拡張」の仕組み**

Claude Codeは、ターミナル上で動くAIコーディングアシスタントです。ChatGPTのような画面ではなく、エンジニアが使う黒い画面の中で動きます。普通のAIチャットとの違いは、あなたのPC上のファイルを直接読み書きできること。コードを提案するだけでなく、ファイルの作成・編集・テスト実行まで全部やってくれます。

そして、Claude Codeには以下の6つの拡張機能があります。

- **CLAUDE.md（メモリ）**：プロジェクトのルールや規約を書いておくファイル。Claude Codeはセッション開始時にこれを読み込んでくれるので、毎回「うちのプロジェクトは…」と説明し直す必要がなくなります。いわば「AIへの申し送り書」。
- **Commands（コマンド）**：よく使う指示をテンプレート化する仕組み。`.claude/commands/review.md` というファイルを作っておけば、`/review` と打つだけでレビューが走ります。
- **Subagents（サブエージェント）**：メインのClaude以外に「専門AI」を立ち上げる仕組み。「設計担当」「テスト担当」「レビュー担当」をそれぞれ独立した環境で動かせます。
- **Skills（スキル）**：再利用できる知識パック。Commandsが「やることリスト」なら、Skillsは「教科書」。Claudeが状況に応じて勝手に参照してくれます。
- **Hooks（フック）**：特定のイベント（ファイル変更やコミットなど）が起きたとき裏側で走る自動処理。
- **Plugins（プラグイン）**：上記をまとめてパッケージ化し、チームやコミュニティで共有する仕組み。

**1. プロンプトの書き方：「何を言うか」で出力が変わる**

Claudeに「自己チェック」させる：
```
この変更をグリルして（厳しくチェックして）。
テストに合格するまでPRは作らないで。
```

「仕切り直し」のプロンプト：
```
ここまでの全情報を踏まえて、今のやり方を捨てて、
エレガントな方法で実装し直して。
```

マイクロマネジメントしない：バグを報告するときは「エラー内容を貼って『直して』と言うだけ」でOK。手順を細かく指示するとClaudeの判断力を制限してしまうことがある。

**2. 計画の立て方：「いきなり書かせない」が鉄則**

- プランモードを使う：`shift + tab` で切り替えると、Claudeはコードを書かずに「計画だけ」を出力してくれます。
- Claudeにインタビューさせる：「以下の要件を読んで、わからないところを質問して。全部答えたら仕様書としてまとめて。」
- フェーズに分ける：大きな機能を一気に作らせるのではなく、段階を区切ってテストを設定。
- もう1人のClaudeにレビューさせる：サブエージェントを使って計画に穴がないかチェック。
- 仕様書は詳しく書く：「ログイン機能を作って」より「メールアドレスとパスワードでログインする機能。パスワードは8文字以上。ログイン失敗時はエラーメッセージを表示。3回失敗でロック」のほうが出力精度が格段に上がります。
- `/simplify` で整理してから次へ：フェーズが終わるたびにコードをリファクタリング。

**3. CLAUDE.mdの育て方：AIを「自分仕様」にする**

何を書けばいいか：プロジェクトの概要、チームのルール、コーディング規約、設計方針の決定事項。

例：
```markdown
# プロジェクト概要
ECサイトのバックエンド。TypeScript + Express で構築。

# コーディング規約
- 変数名はキャメルケース（例: userName）
- APIキーは環境変数から取得。ソースコードに直書き禁止

# よく使うコマンド
- テスト: npm test
- 開発サーバー: npm run dev
```

- 毎日更新する：セッション終了時にCLAUDE.mdを更新する習慣をつけると、翌日のClaudeが「昨日の続き」をスムーズに引き継いでくれます。
- `@`記法で外部ファイルを読み込む：CLAUDE.mdに `@docs/api-spec.md` と書いておけばClaudeがそのファイルも一緒に読んでくれます。
- ルールをファイルで分割する：`.claude/rules/` というフォルダにルールファイルを分けて置けます。
- auto-memoryを使う：Claudeがメモリを勝手に更新してくれるモードもあります。
- lessons learnedを記録する：Claudeが過去に犯したミスのパターンをCLAUDE.mdに書いておくと、同じミスの再発を防げます。

**4. デバッグのコツ**

- 情報は全部渡す：エラーのトレースやログは省略せずに全部渡す。
- まず再現、それから修正：「まずこのバグを再現して、それから修正して」と頼む。
- 根本原因を聞いてから直す：「このバグの根本原因を説明してから、修正案を出して。」
- 失敗したら巻き戻す：`Esc` キーを2回押すか、`/rewind` コマンドで、変更前の状態に戻せます。
- Webアプリは `/chrome` でデバッグ：ブラウザを起動して、画面の表示を確認しながらデバッグ。
- エッジケースを確認させる：「想定外の入力パターンでもテストして」と頼む。

**5. Commandsの活用：繰り返しをなくす**

- よく使う指示をファイルにする：`.claude/commands/review.md` にレビュー用のプロンプトを書いておけば、`/review` で一発実行。
- 役割別にコマンドを作る：フロントエンド開発用、バックエンド開発用、ドキュメント作成用など。
- コマンドを連鎖させる：「まずレビュー → 問題なければテスト → 通ったらPR作成」のようにチェーンで使う。
- 日常チェックをコマンド化する：`/daily-check` のようなコマンドを作って、毎朝プロジェクトの基本動作を確認。
- コマンドにサンプルを入れる：「こういう形式で出力して」という例を入れておくと出力の品質が安定します。

**6. Subagentsの使い方：チームで働かせる**

- 役割を明確にする：`.claude/agents/reviewer.md` のような定義ファイルに、役割・使えるツール・使えないツールを書いておく。
- 独立したコンテキストを与える：「エージェントには新しい独立したコンテキストを与えて、コンテキスト汚染を防ぐ」ことを強調。
- Agent Teamsで並列に走らせる：複数のサブエージェントを同時に立ち上げて、独立したブランチで並列作業。「3つの機能を実装して」→ 3エージェントが同時作業 → 完了後にマージ。
- 独立して動作確認してから組み合わせる。

**7. Skillsの活かし方：知識を再利用する**

- Commandsとの使い分け：Commandsは「自分で `/名前` で呼び出す」もの。Skillsは「Claudeが状況に応じて勝手に参照する」もの。
- 領域ごとに作る：React用、テスト用、API設計用などドメインごとに分けると管理しやすい。
- 段階的に情報を開示する：スキルの中身を最初から全部読み込ませるとコンテキストウィンドウを消費するため、「必要になったときに必要な部分だけ読み込む」書き方にする。

**8. ワークフローの組み方：全部をつなげる**

Command → Agent → Skill パターン：`/implement` と打つだけで、architectが設計 → implementerが実装 → reviewerがレビューまで一気に回る。

「Research → Plan → Execute → Review → Ship」の流れ：
1. 調査（Research）で今のコードの状態を把握
2. 計画（Plan）でプランモードを使って方針を立てる
3. 実行（Execute）で計画に沿ってコードを書く
4. レビュー（Review）でサブエージェントにチェックさせる
5. リリース（Ship）でテストを通してPRを作成

- Git Worktreeで並列作業：同じプロジェクトの別ブランチを同時に作業できるGitの機能で、複数のエージェントが独立した環境で同時に作業できます。
- `/schedule` でクラウド実行：PCの電源が入っていなくても、クラウド上にタスクを登録して定期実行させられます。
- `/loop` でローカル連続実行：自分のPC上で最大3日間の連続実行ができます。

**9. Git / PRの運用**

- PRは小さく保つ：1つのPRには1つの変更が基本。
- コミットメッセージはClaudeに書かせる：変更内容を理解したうえで適切なメッセージを生成してくれます。
- マージ前にClaudeにチェックさせる：「mainブランチとの差分を確認して、互換性を壊す変更がないかチェックして」と頼む。
- PRのレビューもAIにやらせる：GitHub Appを使うと、PRが作られたタイミングでClaude Codeが自動でレビューしてくれます。

**10. 毎日の運用**

- フォルダ構成を統一する：一貫した構成にしておくほど精度が上がります。
- Claudeの出力は必ず目視確認する：AIは間違えることがあるので、最後のチェックは人間が担います。
- Auto Modeで許可確認を省略する：起動時に `--permission-mode auto` をつけると、ファイル書き込みやコマンド実行のたびに「許可していいですか？」と聞かれなくなります。
- 音声入力を使う：`/voice` コマンドで音声入力モードに切り替えられます。日本語にも対応。
- Remote Controlでどこからでも操作：`/remote-control` を使うと、PCで始めたセッションをスマホやタブレットから操作できます。

**11. 有名な開発ワークフローの紹介**

- Superpowers（スター数12.2万）：テスト駆動開発を最重視するスタイル。https://github.com/obra/superpowers
- Spec Kit（スター数8.3万）：仕様書駆動のアプローチ。コードを書く前に詳細な仕様書を作り、Claudeはその仕様書に忠実に従って実装。https://github.com/github/spec-kit
- Get Shit Done（スター数4.4万）：実用重視のスタイル。タスクごとに新しいコンテキストを使い、複数タスクを並列で一気に片付けます。https://github.com/gsd-build/get-shit-done

**まとめ**

87個のTipsを紹介したが、「全部やらなきゃ」とは思わなくて大丈夫です。まずはCLAUDE.mdを作る。次にプロンプトの書き方を意識する。慣れてきたらCommandsを1〜2個作って、SubagentsやSkillsに手を出してみる。Claude Codeは「使い込むほど、自分のプロジェクトに最適化されていくツール」です。

### トークン代が半分になった：Claude Code × Pythonの「役割分担」設計術

- 投稿者：@sarukun99（引用元：@kawai_design）
- 日時：2026-03-31 18:36:59
- URL：https://x.com/sarukun99/status/2039049332884984098

**ツイート本文（引用元ツイート全文）：**

（引用ツイート：https://x.com/kawai_design/status/2038958744852394412）

**Claude Codeに全部やらせていた頃の話**

「AIに全部やらせれば楽じゃん」
私もそう思っていました。ニュースの収集、記事の執筆、サイトの更新。全部Claude Codeに任せれば、寝ている間に仕事が終わる。
実際にやってみました。
Claude Codeにニュース収集を1日7回やらせ、毎回WebSearchで情報を探させ、記事を生成させ、デプロイまで一気通貫。
結果、動きました。ちゃんと動きました。
ただ、1つ問題がありました。トークン消費が想定の2倍以上だったのです。
冷静に考えれば当たり前です。RSSフィードの巡回なんて、同じURLを同じ形式で叩くだけ。判断力は要りません。それなのに毎回Claude Codeを起動し、WebSearchでニュースを探させていた。
高級シェフに皿洗いをさせていたようなものです。

**判断基準はたった1つ。「ロジックが固定か、毎回判断が要るか」**

この失敗から見つけた使い分けの原則は、驚くほどシンプルです。
- ロジックが固定 → Python。
- 毎回判断が変わる → Claude Code。

Pythonに任せるもの（固定ロジック）：
- RSS/APIからのデータ収集
- CSV/JSONの整形・集計・フィルタリング
- ファイルの一括リネーム・移動・変換
- 定時実行（launchd/cron）
- 重複排除・テーマ分類（キーワードマッチング）

Claude Codeに任せるもの（判断が必要）：
- 「どのニュースが重要か」の選別
- 記事・投稿文の生成
- 複数ファイル横断の編集
- 「今日何をやるべきか」の優先度判断
- 新しいワークフローの設計・試行錯誤

**実際に作ったパイプラインの全体像**

Before：Claude Code単独（1日7回起動）
```
Claude Code (WebSearch) → 記事生成 → デプロイ × 7回/日
```

After：Python + Claude Codeの3フェーズ（1日3回起動）
```
Phase 1: Python（トークン0）
  RSS収集 → テーマ自動分類 → 重複排除 → raw_news.json

Phase 2: Claude Code（判断のみ）
  raw_news.json読み → ランキング → 角度決定 → atoms.csv転記

Phase 3: Claude Code（生成のみ）
  新規atoms → 記事JSON生成 → サイトビルド → デプロイ → git push
```

**シェルスクリプトが「橋渡し役」になる**

```bash
# ai-news-collect.sh（簡略版）

# Phase 1: Python（トークン0）
python3 brain_content.py rss --days 1

# ニュースが0件なら終了（Claude Code起動しない = コスト0）
if [ "$NEWS_COUNT" -eq 0 ]; then exit 0; fi

# Phase 2: Claude Code（判断）
claude -p "raw_news.jsonを読んでランキング+atoms.csv転記"

# Phase 3: Claude Code（生成 + デプロイ）
claude -p "新規atomsからGuide投稿を生成 → build → deploy"
```

シェルスクリプトの役割は3つ：
- ゲートキーパー：Pythonの結果が0件ならClaude Codeを起動しない（無駄なトークン消費を防ぐ）
- コンテキスト分離：Phase 2とPhase 3を別セッションにし、各フェーズの責務を明確にする
- ログ管理：各フェーズの開始・終了をログに記録

**この設計で得られた具体的な効果**

- トークン消費：1日7回 × WebSearch込み → 1日3回 × 整理済みデータ読み。体感で半分以下に。
- アウトプット：同じパイプラインから、SEO記事・X投稿・note記事・Instagram投稿にも展開可能
- 信頼性：RSSフィードという確実なソース + Claude Codeの判断力。的外れなニュースを拾わなくなった。

**あなたが今日できること**

1. Claude Codeに「この作業をPythonスクリプトにして」と頼む
2. そのPythonスクリプトをcron/launchdで定時実行する
3. 結果をJSONやCSVに吐き出す
4. Claude Codeはそのファイルを読むところから始める

Claude Codeは優秀なシェフです。でもシェフに皿洗いはさせない。仕込みはPythonに任せて、Claude Codeには「味付け」に集中してもらう。それが、私がたどり着いた最適解です。

### Claude Codeで1週間に自動化できたこと

**ツイート本文：**

もうClaude Codeで自動化できないものないんじゃないかな。

（引用ツイート：https://x.com/L_go_mrk/status/2039298726574088496）

**Claude Codeで1週間で自動化できたこと**

「Claude Codeで何ができるの？」とよく聞かれます。正直、僕も最初は「コード書くやつでしょ？」ぐらいの認識でした。非エンジニアですし。ただ、この1週間で、事業運営の中で「地味に時間を食ってた作業」がかなり自動化できました。今回はその中身を全部晒します。何を自動化して、どうやって作って、どれぐらい時間が浮いたか。

**① ナレッジ管理の自動化**

これが一番デカいです。AI駆動塾では、海外のAI情報・X投稿・note記事のネタ・メンバーからのフィードバックなど、あらゆる情報をナレッジとして蓄積しています。今、ナレッジノートは450件以上あります。以前は「いい情報を見つけたらとりあえずブクマ」だけして必要になったら引っ張り出す（覚えていないこともしばしば）。正直全く意味がありませんでした。

今はSkillGraphsという考えを導入して、Claude Codeに16個のSkillを作って、この流れを自動化しています。
- `/取り込み` — 素材をキャプチャフォルダに放り込むだけ。重複チェックも自動
- `/分解` — 1つの素材から、使える知見を1件ずつ抜き出してナレッジノートにする
- `/接続` — 新しいノートと既存の450件以上のノートの接続を自動で発見する
- `/検証` — ノートの品質チェック（タイトルが曖昧じゃないか、リンク切れがないか等）
- `/パイプライン` — 上の全工程を一気通貫で実行

例えば海外の長文記事を1本処理する場合、以前は読む→理解する→分類する→メモするで1〜2時間。今は「/パイプライン [ファイル名]」で15分です。しかも接続の質が手動より高い。人間が気づかない「3ヶ月前に書いたあのノートと繋がる」みたいな接続をClaudeが見つけてくれます。

**② 海外AI情報のリサーチ自動化**

AI駆動塾のX投稿で最もインプレッションが高いのは、海外のAI情報を日本語で紹介する投稿です。1投稿で最大192人のフォロワーを獲得した実績があります。問題は、海外情報のリサーチが地味にしんどいこと。Reddit、Hacker News、X、YouTube、TikTok…ソースが多すぎます。

これを `/last30days` というSkillで自動化しています。「/last30days Claude Code最新動向」と打つだけで、10以上のソースから直近30日の情報を横断検索して、引用元付きのレポートにまとめてくれます。以前はRSS読んで、X巡回して、Reddit覗いて…で毎朝1時間。今は5分でレポートが出てきます。

**③ X投稿の作成支援**

投稿そのものをAIに丸投げはしていません。最終判断は必ず自分でやります。ただ、「素材集め → 構成 → 下書き」の部分はClaude Codeにかなり助けてもらっています。
- ナレッジグラフから関連する知見を引っ張ってきて、投稿の素材にする
- 海外投稿を翻訳して、日本語の投稿フォーマットにリライトする
- 過去にウケた投稿パターンに当てはめた構成案を出す

ポイントは、450件以上のナレッジノートがあるからこそ、投稿に使える素材がすぐ出てくるところです。ナレッジが貯まれば貯まるほど、投稿の引き出しが増える。この複利構造が、①のナレッジ管理を真面目にやっている最大の理由です。

**④ セッション管理の自動化**

Claude Codeはセッション（会話）を切り替えるたびにコンテキスト（記憶）がリセットされます。これをHooks（自動実行スクリプト）で解決しています。
- セッション開始時 — 自動的にプロジェクトの現在の目標、進行中の作業、未処理タスクの数をClaudeに読み込ませる
- セッション終了時 — 今のセッションで何をやったか、次に何をすべきかを自動で記録する
- ノート作成時 — 品質チェック（スキーマ検証）を自動で走らせる
- ノート保存時 — Gitで自動コミット。変更が消えない

つまり、ターミナルを開いた瞬間にClaudeが「今日の状況」を把握していて、閉じた瞬間に「今日やったこと」が記録される。毎回ゼロスタートにならない仕組みです。

**⑤ グラフ分析**

450件以上のナレッジノートがウィキリンクで繋がっていると、「どのノートが孤立しているか」「どのノートが情報のハブになっているか」「まだ繋がっていないけど繋がるべきノートはどれか」が見えてきます。これを `/分析` というSkillで可視化しています。
- 孤立ノート検出 — どこからもリンクされていないノートを発見
- ブリッジ検出 — 複数の領域をつなぐ構造的に重要なノートを特定
- 三角形検出 — A→B、B→C、でもA→Cがない、みたいな「合成の機会」を発見

**どうやって作ったか**

全部「非エンジニア」の僕が作りました。最初から16個のSkillがあったわけではないです。順番としては：
1. まずCLAUDE.mdにプロジェクトの基本ルールを書いた
2. 「毎回同じことを説明してるな」と思った作業をSkillにした
3. 「Skillを実行した後に毎回やる確認作業」をHooksにした
4. 使いながら「ここ微妙だな」と感じたらSkillを修正した

大事なのは「一気に作ろうとしない」ことです。摩擦を感じたところだけ自動化する。最初の1週間は5つぐらいしかSkillがなかった。それで十分回っていました。

**自動化して変わったこと**

一番変わったのは「情報を貯めるのが苦じゃなくなった」ことです。以前は情報収集が「やらなきゃいけないこと」でした。今は素材を放り込むだけでナレッジになるので、インプットのハードルが激減しました。結果として、投稿の引き出しが増えて、記事の質が上がって、フォロワーが増えて、メンバーシップの加入者が増えた。自動化の効果が複利で回り始めています。

### Cowork と Claude in Chrome、結局どっちを使うべき？【初心者向け】

- 投稿者：@ai_jitan（引用元：@dansyu_callenge）
- 日時：2026-03-31 22:56:17
- URL：https://x.com/ai_jitan/status/2039114588458000691

**ツイート本文（引用元ツイート全文）：**

（引用ツイート：https://x.com/dansyu_callenge/status/2039085713413865680）

意外と知られてないんだけど、Cowork と Claude in Chrome、使い分けを間違えると枠が一瞬で尽きます。
Claudeを使っていて、こう思ったことはありませんか。「あれ、今日やけに制限が早いな」
僕も最初、そうでした。原因は、Claude in Chrome と Cowork の違いが分からなくて、全部 Cowork に投げていたからです。

**まず結論。Cowork は"PC内の秘書"、Claude in Chrome は"ブラウザの助手"です**

いちばん大事なことを、最初に言います。
- Claude Cowork は「PC内のファイルや連携ツールを操る優秀な秘書」です。
- Claude in Chrome は「Webブラウザを横で一緒に見て、代わりに手を動かしてくれる助手」です。

Cowork は、ローカルのフォルダやファイルを直接操作します。PDFを読んで、Excelにまとめる。バラバラのファイルを整理する。複数のツールを連携させる。PC内のデータやファイルを扱う作業に強いです。

Claude in Chrome は、ブラウザの中で動きます。Webページを読んで、情報を抽出する。フォームに入力する。複数のタブをまたいで比較する。ブラウザ上の作業を代行してくれます。

**Claude Cowork の体験：「データやファイルを裏側で一気に処理する」**

Claude Cowork は、Claude Desktop アプリ内で動作します。PC のローカルファイルや外部ツール(Jira、Google ドライブなど)を直接操作する業務エージェントです。操作の始まりは、デスクトップアプリを開き、作業対象の「ローカルフォルダ」を指定して指示を出します。
- 「このフォルダにある数十枚のPDF履歴書からスキルを抽出してExcelにまとめて」
- 「バラバラの議事録ファイルの日付を統一して整理して」

こういった指示を出すと、AIが事前に「何をどう処理するか」という作業計画を提示してきます。ユーザーがそれを確認・承認(実行)すると、AIが裏側で自律的にファイルを読み書きし、処理を完了させます。

体験のポイントは、大量のファイル処理やデータ集計、複数ツールの連携など、手作業では時間がかかる「データ操作」を、計画の承認だけで一気に終わらせてくれることです。毎朝「このテーマでリサーチして、資料をまとめて」と頼むと、1時間後には資料ができています。手動でやったら3時間かかる作業です。

**Claude in Chrome の体験：「Web画面を見ながら手作業を代行してもらう」**

Claude in Chrome は、Chrome ブラウザの拡張機能としてサイドパネルに常駐します。
- 「この Web フォームに以下の担当者情報を入力して」
- 「開いている3つのタブの製品情報を比較表にして」

こういった指示を出すと、AIが実際に Web ページを順番に訪問したり、フォームの各項目を認識して入力したりする様子を、実際のブラウザ画面を見ながら確認できます。3つのサイトを開いて、「この3つの違いをまとめて」と頼むと、5分で終わります。手動でやったら30分かかる作業です。

**具体的なケース別使い分け**

- ケース1：競合サイトをいくつか見て比較したい → **Claude in Chrome**（複数タブを開いて比較）
- ケース2：ローカルのファイルを整理して、レポートにまとめたい → **Cowork**（ローカルファイルを直接扱える）
- ケース3：Webページを見ながら何かを入力したい → **Claude in Chrome**
- ケース4：データを集計して、複数のツールを連携させたい → **Cowork**

**それぞれの"隠れたメリット・デメリット"**

Claude in Chrome のメリット：ブラウザ上の作業に入るまでが早い。見えているページをそのまま扱えるのが強い。
Claude in Chrome のデメリット：ブラウザの外にある仕事全体までは持ちにくい。通常チャットの2〜3倍消費します。ページの情報量が多いほど、消費が跳ねます。

Cowork のメリット：まとまった仕事を任せやすい。定期タスクや、ファイルを含む作業との相性が良い。
Cowork のデメリット：消費も重くなりやすい。簡単な作業にはオーバースペックになりやすい。通常チャットの3〜5倍消費する傾向があります。

**僕が実際にどう使い分けているか**

「ブラウザで完結するなら Claude in Chrome、それ以外なら Cowork」と決めてから、枠の減りが明らかに遅くなりました。そして、2つを組み合わせることもあります。「Claude in Chrome に指示して競合サイトから最新データを収集させ、その情報を Cowork に渡してローカルの Excel にレポートとしてまとめさせる」といった流れです。Web 調査から資料作成までの一気通貫の自動化が可能になります。

**どんな人がどっちを使えばいいのか**
- ブラウザ作業が中心の人 → Claude in Chrome
- ローカルのファイルやデータを扱う人 → Cowork
- まず軽く使いたい人 → 通常チャット
- 開発やファイル編集寄りの人 → Claude Code

**迷ったらどう選ぶか**

迷ったら、まず自分の作業が「ブラウザ内で完結するかどうか」で考える。
- ブラウザ内で完結するなら、Claude in Chrome。
- ブラウザの外にある作業を含むなら、Cowork。

### Claude CodeでX自動投稿するロードマップ — AI分身の作り方

- 投稿者：@akaoniudetate
- 日時：2026-04-02 12:00:29
- URL：https://x.com/akaoniudetate/status/2039674327604576314

**ツイート本文：**

これは、、、もうみんなX自動化できるやん、、、

**引用ツイート全文（https://x.com/akaoniudetate/status/2039627365555552556）：**

**Claude CodeでXを自動投稿するロードマップを教えます**

正直、この話は出すか迷った。数ヶ月前、ある実験を始めた。「自分がXを開かなくても、自分の言葉で、自分の価値観で、投稿され続けたらどうなるか？」少なくとも自分が見た範囲では、明確にAI投稿だと見抜かれる反応はほとんどなかった。それどころか、「最近キレてますね」「刺さりました」とリプが来た。

書いたのはAI。ただのAIじゃない。俺の過去投稿を読み込ませて、口調も、価値観も、伸びるパターンも分析させた「もう一人の自分」これを、ここでは **AI分身** と呼ぶ。

**重要な前提**
- これは「本人の発信スタイルを補助的に再現する運用」の話
- 第三者へのなりすまし、他人の文体模倣、虚偽の本人性を装う運用は避けるべき
- 完全放置の自動化ではなく、まずは **生成 → 人間レビュー → 投稿** の流れで始める前提で読んでほしい
- 認証切れ、API変更、投稿失敗、ポリシー変更の監視は必要

**なぜ普通のAI投稿は違和感が出やすいのか**

ChatGPTやClaudeにそのまま「Xの投稿を考えて」と頼むと、確かにそれっぽい文章は出てくる。でもタイムラインに流れた瞬間、違和感が出やすい。理由は単純で、あなたのクセが入っていないから。人は文章の中身だけを読んでいるわけじゃない。語尾の選び方、句読点の打ち方、改行のリズム、毒の入れ方、結論までの持っていき方。そういう「言語化しにくい空気感」で、そのアカウントの人格を判断している。だから、「AIに投稿を書かせる」だけでは足りない。やるなら、AIを自分に寄せてから書かせる必要がある。

**AI分身の正体（仕組みはシンプル）**

1. 自分の過去投稿を集める
2. その投稿から、自分の口調や価値観、構成パターンを分析させる
3. 分析結果をもとに「自分っぽく書くためのシステムプロンプト」を作る
4. そのプロンプトを使って新しい投稿文を生成する
5. X API経由で、レビュー後に投稿する

ここで使うのが Claude と Claude Code だ。Claudeは「文体分析と投稿生成の脳みそ」。Claude Codeは、その流れを実行するためのスクリプトや実行環境を組み立てる役割を持つ。

**全体マップ（6STEP）**

1. X APIの認証情報を取得する
2. Anthropic APIキーを取得する
3. Claude Codeをインストールする
4. 自分の過去投稿を集める
5. Claude CodeでAI分身の土台を作る
6. 下書き生成から始めて、必要に応じて定期実行へ広げる

所要時間の目安：慣れている人なら短時間で進められるが、初回は開発者登録、課金設定、環境変数の設定、スケジューラ設定まで含めて **1〜3時間以上** は見ておいたほうが安全。

**用意する中心ツール（4系統）**

1. Claude（https://claude.ai/）：投稿の分析と生成を担当するAI本体。
2. Claude Code（https://docs.anthropic.com/en/docs/claude-code）：自動化スクリプトを作るための土台。
3. Anthropic API（https://console.anthropic.com/）：Claudeをプログラムから呼び出すためのAPIキー。
4. X API（https://developer.x.com/）：生成した投稿を実際にXへ送るためのAPI。

**手順④：自分の過去投稿を集める**

AI分身の精度は、ここでかなり決まる。雑に集めると、出てくる投稿もぼんやりする。
- 自分のXプロフィールを開き、反応が良かった投稿を中心にピックアップ
- 最低30投稿、理想は50〜100投稿
- 投稿本文を1つずつコピーして、テキストファイル（my_posts.txt）に保存

**手順⑤：Claude CodeでAI分身の土台を作る**

Claude Codeに指示する内容（文体分析＋システムプロンプト作成）：
```
このフォルダにある my_posts.txt を読み込んで、以下の分析をしてください。
1. 口調の特徴（語尾の傾向、一人称、句読点の使い方、改行のクセ）
2. よく使うフレーズやワード
3. 投稿の構成パターン（問題提起→主張、体験談→教訓、など）
4. 扱っているテーマの傾向
5. 反応が取れそうなフックの型
分析結果をもとに、「この人物になりきってXの投稿を生成するためのシステムプロンプト」を作成してください。
このシステムプロンプトを system_prompt.txt として保存してください。
```

続けてスクリプト作成を指示：
```
以下の仕様でPythonスクリプトを作成してください。
【機能】
- system_prompt.txt を読み込み、それに基づいてXの投稿文を生成する
- 生成にはAnthropic APIのClaudeを使う
- 生成した投稿文をX API v2を使って自動投稿する
- 投稿前に dry_run モードで投稿内容をコンソールに出力する機能をつける
- 過去の投稿ログをファイルに保存し、類似度の高い内容は投稿しないチェック機能をつける

【投稿文の生成ルール】
- 140文字以内（日本語）
- system_prompt.txt の人格・口調・構成パターンに従う
- 毎回異なるテーマ・切り口で生成する
- AIっぽい表現（「〜ですね」「いかがでしょうか」など）は使わない

【ファイル構成】
- main.py（メインスクリプト）
- requirements.txt（必要なライブラリ一覧）
- .env.example（環境変数のテンプレート）
- .gitignore（.envをGit管理から除外する設定）
```

**dry runで確認**

```bash

```

コンソールに出た投稿文を見て確認：
- 自分の文体に近いか
- 不自然な言い回しがないか
- 危ない表現が混ざっていないか
- ポリシー違反の恐れがないか

問題がないと判断できてから本番実行。最低でも最初の数十本は dry run → 人間レビュー → 投稿の流れを続けることを推奨。

**Xの自動化ポリシーに関する注意**

特に注意したい点：

- 自分の言葉がある人ほど強い
- 過去投稿が多い人ほど精度が上がる
- 何も蓄積がない状態では、出力も浅くなりやすい

50個のうち自分が手を動かしてるのは「最終チェック」と「GOサインを出す」だけ。作業の実行はClaude Codeがやって、自分は判断だけする。これが「AIを使いこなす」の実態だと思ってます。

ポイントは3つ：
1. まず自分の仕事を「手順」として書き出す（言語化）
2. その手順をスキルとして保存する（再利用）
3. スキルを組み合わせてパイプラインにする（自動化）

560個試して残ったのが40個。50個の自動化を支えてるのがその40個のスキル。「作ること」より「捨てる判断」の方がよっぽど難しかった。
