## ▼ 現在の構成（確定版・2026-04-11時点）

```
【SessionStart】
  setup.sh        → シンボリックリンクを自動確認・作成
  gdrive-download.sh → basic-memory DB を Google Drive からダウンロード

【write_note するたびに（PostToolUse）】
  gdrive-upload.sh → 書いた瞬間に Google Drive へアップロード

【SessionStop】
  gdrive-upload.sh → セッション終了時にアップロード

【「記憶して」と言うと（UserPromptSubmit）】
  system-reminder に write_note を促す指示が届く
  → Claude が mcp__basic-memory__write_note で保存

【記憶を参照するとき（CLAUDE.md ルール）】
  1. auto-memory：~/.claude/projects/【パス】/memory/MEMORY.md を読む
  2. basic-memory：recent_activity および関連ノートを read_note で読む
```

### 同期対象ファイル（Google Drive）

| ファイル | 用途 |
|---|---|
| basic-memory の SQLite DB | 全記憶ノート（セマンティック検索対応） |
| processed_ids.json（v1/v2） | Xブックマーク処理済み ID |

### 使用ツール

| ツール | 役割 |
|---|---|
| basic-memory | MCP サーバー（SQLite + FastEmbed） |
| rclone | Google Drive との同期 |
| systemd フック | Claude Code のイベント駆動処理 |
| dotfiles + シンボリックリンク | 設定の PC 間統一管理 |

## ▼ 構築を通じて得た設計判断

1. **「行動ルール」は CLAUDE.md、「トリガー」はフック** — フックで呼び出しを制御しようとしたが誤発火が発生。役割を分離することで安定した。

2. **シンボリックリンクが全体の基盤** — settings.json・CLAUDE.md・scripts のどれか一つでも古いと全体が壊れる。シンボリックリンクによる一元管理が前提。

3. **書いた瞬間に同期** — セッション終了時のアップロードだけでは不十分。write_note の PostToolUse フックを追加することで即時同期を実現。

4. **Windows と Linux の差異への対処** — シンボリックリンクの仕様差（ディレクトリは管理者権限が必要）を `mklink /J`（ジャンクション）で解決。
