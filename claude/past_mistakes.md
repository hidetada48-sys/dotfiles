### M-001 Edit tool: 重複文字列で置換失敗（2026-05-07）
**状況：** `post-tool-check.sh` に `exit 0` が3箇所あり、replace_all=false で Edit を実行したらエラー。  
**やらかし：** ファイル内に同じ文字列が複数あることを確認せずに Edit を呼んだ。  
**対策：** Edit 前に grep で対象文字列の出現数を確認する。ユニークにするため前後の文脈を必ず含める。

### M-008 調査前に断言し、指摘一言で翻した（2026-05-09）
**状況：** Windows互換チェックを求められた際、setup.shも読まずに「致命的な問題あり」と断言。ユーザーの一言で「問題ない」に翻し、その後にsetup.shを確認した。  
**やらかし：** 調査の順序が逆。コードを読む前に判断を出し、証拠なく断言した。指摘されて翻したことで判断の信頼性がゼロになった。  
**対策：** 互換性・動作の判断は必ずコード・記憶・設定ファイルを読んでから出す。「たぶんこう」で断言しない。調査→判断の順を崩さない。

### M-009 複雑なBash・巨大出力でツール呼び出しがparse失敗→数分ハング（2026-06-07）
**状況：** 「固まった」とユーザーが体感した現象の一つ。`grep -n -iE "複雑な正規表現" | tail`、`bc`・`date +%s.%N`・forループなどを1行に詰めたBashを投げると "The model's tool call could not be parsed (retry also failed)" が出て、リトライも同じ複雑さで連続失敗し2〜4分ハングする。調査中に自分で何度も再現させた。
**やらかし：** ツール呼び出し（特にBash）に長い引数・特殊文字（`|` `"` 正規表現 複数リダイレクト）・巨大出力を生む指示を詰め込んだ。ツール呼び出しのJSON組み立てが壊れる。
**対策（必ず守る）：**
1. Bashは1コマンド1目的に割る。パイプ連結・複雑正規表現・複数リダイレクトを1行に詰めない。
2. 大量出力を出さない。`grep`は`-c`や対象行を絞る。`| tail`で巨大出力を生む形にしない。全文検索はGrep/Readツールか範囲指定`sed -n`を使う。
3. 生Bashの巨大grepより専用ツール（Read/Grep）を優先する。
4. Bash限定ではない。Edit/Write の content・new_string も同じ。日本語長文・表（パイプ）・括弧・コードブロックを一度に詰めず、1回の編集を小さく刻む。大きな置換は数回に分ける。
（補足：これとは別に、ツール無しの長文生成が遅いだけの「固まり」もある。原因はサーバー側の生成速度ムラで制御不能。被害を減らすには出力を短くする＝一度に大量の情報を出さない。）
