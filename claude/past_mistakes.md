### M-009 複雑なBash・巨大出力でツール呼び出しがparse失敗→数分ハング（2026-06-07）
**状況：** 「固まった」とユーザーが体感した現象の一つ。`grep -n -iE "複雑な正規表現" | tail`、`bc`・`date +%s.%N`・forループなどを1行に詰めたBashを投げると "The model's tool call could not be parsed (retry also failed)" が出て、リトライも同じ複雑さで連続失敗し2〜4分ハングする。調査中に自分で何度も再現させた。
**やらかし：** ツール呼び出し（特にBash）に長い引数・特殊文字（`|` `"` 正規表現 複数リダイレクト）・巨大出力を生む指示を詰め込んだ。ツール呼び出しのJSON組み立てが壊れる。
**対策（必ず守る）：**
1. Bashは1コマンド1目的に割る。パイプ連結・複雑正規表現・複数リダイレクトを1行に詰めない。
2. 大量出力を出さない。`grep`は`-c`や対象行を絞る。`| tail`で巨大出力を生む形にしない。全文検索はGrep/Readツールか範囲指定`sed -n`を使う。
3. 生Bashの巨大grepより専用ツール（Read/Grep）を優先する。
4. Bash限定ではない。Edit/Write の content・new_string も同じ。日本語長文・表（パイプ）・括弧・コードブロックを一度に詰めず、1回の編集を小さく刻む。大きな置換は数回に分ける。
（補足：これとは別に、ツール無しの長文生成が遅いだけの「固まり」もある。原因はサーバー側の生成速度ムラで制御不能。被害を減らすには出力を短くする＝一度に大量の情報を出さない。）

### M-010 WindowsだけJina MCP設定が壊れたまま放置→法改正監視が実行不能（2026-06-16）
**状況：** 週初回フロー④の法改正監視（web調査必須）を実行しようとしたら Jina が使えず停止。調べると Windows の `~/.claude.json` の MCP 設定が `npx -y @jina-ai/mcp-server-jina`（npmに存在しない=404）・サーバー名 `jina-ai`（ルールは `mcp__jina__`）・`env` 空（キー無し）で、起動時にサーバーが立たずツールが1つも登録されていなかった。再起動でも直らない。Linux機は M-003 で「グローバルバイナリ＋キー直書き＋alwaysLoad」と対策済みだったが、その修正が Windows に反映されていなかった（M-005/M-007 と同じマルチOS設定漏れの再発）。
**やらかし：** ① MCP 設定変更後に「両OSでツールが実際に出るか」を確認していなかった。② web必須スキルが、依存ツール未接続でも黙って失敗する設計になっていた（起動フックはレポートファイルの有無しか見ない）。正しいパッケージは `jina-mcp-local`（`search_web`/`read_url` をそのまま公開）。
**対策：**
1. MCP 設定を変えたら、**両OSで再起動し ToolSearch でツールが実際に出るか**を確認してから完了とする（推定で「動くはず」にしない＝M-004）。
2. Jina は OS ごとに `~/.claude.json` の `jina`（サーバー名）= グローバルバイナリ `jina-mcp-local` + `env.JINA_API_KEY` 直書き + `alwaysLoad` で統一。npx は使わない。
3. web 必須スキル（hr-law-watch / intelligence-weekly 等）は冒頭で Jina 未接続を検知したら**黙って失敗せず明確に停止して理由を告げる**。
4. APIキーは平文をルーティン/命令文に埋め込まない。漏洩したら即無効化・再発行（intelligence/SUMMARY の申し送り参照）。

### M-011 古いローカルのdotfilesを監査して「実在しない」と誤断→不要な修正を重ねた（2026-06-16）
**状況：** 法改正監視フック `hr-law-watch-alert.sh` は別セッション（Linux）で実装・push 済みだったのに、Windows ローカルの dotfiles が origin より2コミット遅れたまま。`git fetch` せずその古い状態を「再点検」し、「フックは実在しない・SUMMARY は虚偽」と断言。その誤った前提で新フックを作り、hr/SUMMARY に誤情報を書いて push、最後は rebase 衝突で右往左往した。M-010・記憶 `weekly-review-stale-summary-trap` と同じ stale 罠の再発。
**やらかし：** ① dotfiles 配下を読む前に `git fetch` で origin と比較しなかった（プロジェクトは起動フックが日次 pull するが **dotfiles は自動 pull されず Windows で遅延する**ことを失念）。② その古い状態のまま「実在しない」と断言（M-008 の調査前断言の再発）。③ 結果、不要な作業とトークンを浪費し、誤情報を一度 push した。
**対策：**
1. **dotfiles 配下のファイルを監査・断言する前に必ず `git fetch` し origin と差分を確認**する。behind なら先に pull してから読む。`~/.claude/` はリンク先が dotfiles なので同じ。
2. 「無い／壊れている」と言う前に、ローカルが最新かを必ず疑う。stale を最新と思い込まない。
3. 設定・フックの追加前に、同種のものが既に origin に無いか（別 OS セッションで作られていないか）を確認してから作る。

### M-012 git pull スキップ時にリモート確認せず「未実施」と誤判断（2026-06-27）
**状況：** セッション開始時、未コミットの削除ファイルで自動 pull がスキップ。週初回チェックで業界調査レポートが「ローカルに無い＝未実施」と出た。実際は Windows 側で実施済みでリモートにプッシュ済みだった。  
**やらかし：** pull スキップの耳打ちを見ておきながら、`git fetch` でリモートを確認せずに「未実施」と受け取った。マルチ環境（Linux/Windows 両方で作業）なのにローカルの状態だけで完結させた。  
**対策：**
1. 週初回チェックで「未実施」が出た場合、かつ pull スキップが出ていた場合は、**`git fetch` → `git ls-tree origin/master` でリモートを確認してから**未実施と判断する。
2. マルチ環境では「ローカルに無い ≠ 未実施」。リモートが正。

### M-013 回答をチャットに平文で貼り・生URLを出した（HTML＋リンク付きURL絶対ルール違反）（2026-07-12）
**状況：** 裏方残作業の提示で、①表をチャットに平文で貼る→②生URLを提示→③markdownリンクにするも「リンクになっていない」と、同一セッション内で叱責が連鎖。ルールはCLAUDE.mdと記憶に散在していたが毎回「私の判断」に委ねられ、判断を挟むたびに落ちた。  
**やらかし：** レポート成果物（残作業一覧含む）の中身を平文で貼るのは違反。生URLの羅列も叱責対象。提示の仕方を都度判断していた。  
**対策：**
1. **すべての回答は原則HTML化**。ユーザーが「チャットで答えろ」と明示しない限りチャット本文に中身を貼らない。
2. **提示は必ず `[題名](URL)` のクリックできるmarkdownリンク**。生URL平文は禁止。Windows・Linux 共通のリンク形式に統一（旧xdg-open運用は廃止）。
3. 判断を消すため、HTML化ツール→ローカルサーバURLを `[題名](URL)` で返す、を機械的に実行。グローバルCLAUDE.md「★回答提示の絶対ルール」が正典。
