# note記事自動生成 作業指示書

ユーザーから「note記事を生成して」と言われたら、以下の手順を順番に実行すること。

---

## 前提

- 作業ディレクトリ: `~/test/note_automation/`
- 永続inputファイル（変わらない）:
  - `~/test/note_automation/input/01_selling_know_how.txt` → 売れる記事のノウハウ
  - `~/test/note_automation/input/04_image_style.txt` → 画像のトンマナ指定
- 素材置き場: `~/test/note_automation/input/materials/`（別リポジトリ等から持ち込んだ素材の一時置き場）
- 記事ごとのinput: `~/test/note_automation/input/YYYY-MM-DD/`（記事化を決めたときにユーザーが作成）
  - `02_theme.txt` → 記事テーマ
  - `03_my_info.txt` → 筆者の一次情報
- 記事ごとのoutput: `~/test/note_automation/output/YYYY-MM-DD/`（記事生成時に自動作成）

---

## 手順

### ステップ0：作業フォルダの特定と my_info ドラフト生成

#### 【作業フォルダの特定】

まず `~/test/note_automation/input/` フォルダを確認し、最新の日付フォルダ（例: `input/2026-03-23/`）を特定する。

- 日付フォルダがない場合は作業を中断し、ユーザーに以下を伝える：
  ```
  ~/test/note_automation/input/YYYY-MM-DD/ フォルダがありません。
  記事用のフォルダを作成して 02_theme.txt と 03_my_info.txt を用意してください。
  （input/materials/ に素材がある場合はそちらを参考にしてください）
  ```
- 以降の手順では、特定した日付フォルダを作業inputとして使用する

#### 【my_info ドラフト生成】

`my-info-extractor` エージェントを呼び出す。

```
my-info-extractor を呼び出す
テーマ：[02_theme.txtから読んだ内容]
```

エージェントがJSONLログを読み取り、ドラフトを提示する。**承認を得るまで次のステップに進まないこと。**

### ステップ0.5：有料記事にするか確認する

ユーザーに以下を聞き、**回答を得るまで次のステップに進まないこと**：

```
この記事は有料部分を入れますか？

- 「有料あり」の場合：ノウハウ全15項目・品質評価全項目を適用します
- 「無料のみ」の場合：項目3（無料部分の設計）・4（有料ラインの設定）・5（セールス文）を無視して進めます
```

回答に応じて以降のステップで適用するノウハウ項目・品質評価項目を切り替える：

| モード | 適用するノウハウ項目 | 品質評価で無視する項目 |
|--------|-------------------|----------------------|
| 有料あり | 全15項目 | なし |
| 無料のみ | 1・2・6〜15（3・4・5を除く） | 有料ライン・セールス文・無料部分設計に関する指摘 |

### ステップ1：インプットファイルを読み込む

#### 【画像スタイル参照ファイルのチェック】

まず `~/test/note_automation/input/YYYY-MM-DD/` フォルダ内に `.png` / `.jpg` / `.jpeg` ファイルがあるか確認する。

**画像ファイルが見つかった場合：**

1. 画像を読み込み、以下の観点でスタイルを分析する（被写体・内容・テキストは無視）：
   - 背景色・グラデーション（方向・色値）
   - アクセントカラーとその使われ方
   - レンダリングスタイル（3D CG / フラット / イラスト等）
   - エフェクト（ネオングロー / パーティクル / 被写界深度等）
   - 構図・レイアウト（ゾーン分割 / 奥行き層数等）
   - タイポグラフィの体裁（フォントの重さ・サイズ比・色・配置）
   - 全体の雰囲気・密度感

2. `~/test/note_automation/input/04_image_style.txt` が既に存在する場合は、上書き前にユーザーへ確認を取る：
   ```
   inputフォルダに参照画像（ファイル名）が見つかりました。
   スタイルを分析し、04_image_style.txt を上書き生成します。
   現在の04_image_style.txtは削除されます。よろしいですか？（OK / スキップ）
   ```

3. OKの場合（または04_image_style.txtが存在しない場合）、分析結果を `~/test/note_automation/input/04_image_style.txt` として保存する。
   - **具体的な被写体・オブジェクト（スマートフォン等）は記録しない**
   - 色・エフェクト・レンダリング・構図・タイポグラフィの体裁のみを記録する
   - ステップ5で記事テーマに合うオブジェクトを選ぶ

**画像ファイルがない場合：** 既存の `04_image_style.txt` をそのまま使用する。

#### 【通常インプット読み込み】

以下の4ファイルを読み込む：

- `~/test/note_automation/input/01_selling_know_how.txt` → 売れる記事のノウハウ
- `~/test/note_automation/input/YYYY-MM-DD/02_theme.txt` → 記事テーマ（#で始まる行はコメントなので無視）
- `~/test/note_automation/input/YYYY-MM-DD/03_my_info.txt` → 筆者の一次情報
- `~/test/note_automation/input/04_image_style.txt` → 画像のトンマナ・スタイル指定（上記で生成済みの場合はそちらを使用）

### ステップ2：Web調査の要否を判断し、必要なら実施する

まず以下の基準でWeb調査が必要かどうかを判断し、**ユーザーに宣言してから進むこと**：

**調査不要なケース（スキップしてよい）：**
- `03_my_info.txt` に体験・エピソードが十分に書かれている
- 記事が「開発記録」「体験談」「作ってみた」系で、外部情報を補う必要がない
- Web情報を混ぜると一次情報の純度が下がると判断できる

**調査が有効なケース：**
- テーマの背景知識・統計・トレンドが必要な場合
- 読者が「なぜこれが重要か」を理解するために外部情報が役立つ場合

**宣言の例：**
- 調査する場合：「テーマの背景情報が薄いためWeb調査を行います」
- スキップする場合：「一次情報が充実しているためWeb調査はスキップします」

**調査を実施する場合：**

`seo-researcher` エージェントを呼び出す。

```
seo-researcher を呼び出す
テーマ：[02_theme.txtから読んだ内容]
```

調査結果を受け取り、ステップ3のタイトル・目次設計に必ず反映させる。
**調査が失敗した場合は作業を中断し、ユーザーに報告する。**

### ステップ3：記事の目次を提示し、承認を得る

以下をすべて踏まえて記事構成を考える：

- 売れる記事ノウハウ（必ず全項目反映）
- 筆者の一次情報
- Web調査結果

**【SEOキーワードの選定】**
ステップ2のSEOキーワード調査結果をもとに、タイトルと見出しを設計する。

**タイトルのルール：**
- 公式：`「対象」＋「テーマ」＋「得られる結果」`
- メインキーワードをタイトルの**前半**に入れる
- 文字数は**32文字以内**（検索結果で途切れないよう）
- 競合記事にない角度・切り口を選ぶ

**見出し（H2）のルール：**
- 各H2にキーワードまたは関連語を自然に入れる
- 「〇〇とは？」「〇〇の違いは？」など疑問形も検索意図に合いやすい
- 見出し直後の冒頭文にもキーワードを自然に入れる

**本文冒頭のルール：**
- 記事冒頭100文字以内に結論・キーワードを入れる（AI検索にも引用されやすくなる）

ユーザーに以下の形式で目次を提示し、**承認を得るまで次のステップに進まないこと**：

```
【目次の確認をお願いします】

タイトル案：〇〇〇〇
想定読者：〇〇〇〇
想定検索キーワード：〇〇〇〇

## 見出し1
## 見出し2
## 見出し3
...

この目次でよければ「OK」、変更があればご指示ください。
```

### ステップ4：記事本文を執筆する

目次が承認されたら本文を執筆する。画像タグは挿入しない（本文は純粋な文章のみ）。

【執筆ルール】
1. 売れる記事ノウハウを一つ残らず反映すること
2. 筆者の一次情報を具体的なエピソードとして盛り込むこと
3. 読者が「自分のことだ」と感じられる書き出しにすること
4. 見出しは `##` を使うこと
5. 文字数は3000〜5000文字
6. note向けの読みやすい文体（短い文・改行多め）

### ステップ4.5：記事本文の確認を取る

記事本文が完成したら、ユーザーに提示して確認を得る。
**承認を得るまで次のステップ（画像プラン）に進まないこと。**

提示形式：

```
【記事本文の確認をお願いします】

（記事本文をそのまま出力する）

内容・文体・構成に問題なければ「OK」とお知らせください。
修正がある場合はご指示ください。修正後に再提示します。
```

### ステップ5：画像の提案を提示し、承認を得る

本文が完成したら、まず `~/test/note_automation/input/04_image_style.txt` を読み込む（このステップの前にまだ読んでいなければ）。その上で本文の構成を踏まえて画像の挿入箇所・枚数・内容を提案する。

**承認を得るまで次のステップに進まないこと。**

【サムネイルには以下を必ず含めること】
- メインコピー：記事タイトルをキャッチーに圧縮した言葉（読者が一瞬で意味をつかめるもの）
- サブコピー：1〜2行で補完する言葉
- 構図の方針：左右の役割（Before/After等）・視線の流れ・主要要素3つ以内

【本文画像には以下を含めること】
- 挿入位置（「## 見出し名」の後）
- 表現する概念・状態（具体的なビジュアルイメージ）

```
【画像プランの確認をお願いします】

合計〇枚を予定しています。

1. サムネイル（記事冒頭）
   - メインコピー：〇〇〇〇
   - サブコピー：〇〇〇〇
   - 構図：左＝〇〇（Before）、右＝〇〇（After）、視線は左→右

2. 本文画像①（「## 見出し名」の後）
   - 表現するもの：〇〇〇〇
   - ビジュアルイメージ：〇〇〇〇

...

枚数や内容を変更したい場合はご指示ください。OKであれば「OK」とお知らせください。
```

### ステップ6：JSONファイルに書き出す

**【最初にやること】** `~/test/note_automation/input/04_image_style.txt` を読み込む（ステップ1でまだ読んでいなければ）。以降のプロンプト生成では、このファイルのフィールド値を直接展開して使うこと。抽象的な言い換えは禁止。

本文への画像タグの挿入は行わない。記事本文は純粋な文章のみとする。

承認済みの画像プランをもとに、`~/test/note_automation/output/YYYY-MM-DD/image_prompts.json` に書き出す。（フォルダがなければ作成する）

---

#### 【言語ルール：厳守】

**全プロンプトは日本語のみで記述する。英語プロンプト一切禁止（固有名詞・ブランド名を除く）。**

救済措置（英語が混入してしまった場合）：
- **推奨**：`Place the Japanese text "〇〇" ([English]) at [position]` の形式で日本語を組み込む
- **代替**：日本語訳プロンプトを追記し、Nano Banana Pro には日本語訳のみ使用する

---

#### 【プロンプト生成ルール：9要素】

各画像のプロンプトは以下の9要素で構成する。

**① 冒頭テキスト表示指示（サムネイルは必須）**

```
画像内に『[メインコピー]』という日本語テキストを左上起点で大きく表示。
```

**② 表示環境の指定（サムネイルは必須）**

```
Xのスマホ画面で読まれることを前提に、メインコピーが一瞬で読めるよう2〜3段に分けて強い視線誘導を作る。
```

**③ メインコピー・サブコピー**

ステップ5で承認されたコピーをそのまま使う。

**④ 構図の指定（具体的に）**

- 左右の役割分担（Before/After等）を明示する
- 主要要素は3つ以内に絞ることを明示する
- 視線の流れを明示する（「左から右へ〇〇の流れで見せる」等）

**⑤ 禁止事項の明示**

```
細かい注釈・多数の要素・均等配置・説明図っぽい表現を避ける。
```

**⑥ 色の指定**

`color_system.brand_colors` の値（google_blue・google_red・google_green・google_yellow）を色コードごとそのままプロンプトに展開する。
`layout_principles.structural_anchor` の値をそのままプロンプトに展開する。
`background_colors` の値をそのまま背景色として展開する。

**⑦ テキスト品質の要件（サムネイルは必須）**

`mobile_x_thumbnail_priority.viewing_context` の値をそのまま展開する。
`mobile_x_thumbnail_priority.success_condition` の値をそのまま展開する。

**⑧ フォント・タイポグラフィ**

`typography.font_family.japanese` の値をそのまま展開する。
`typography_for_thumbnail.main_title_position` の値をそのまま展開する。
`typography_for_thumbnail.main_title_layout` の値をそのまま展開する。
`typography_for_thumbnail.subcopy_rule` の値をそのまま展開する。

**⑨ スタイル・トーン・禁止事項**

`information_density.max_major_elements` の値 → 「主要要素は〇個以内に絞る」として展開する。
`information_density.prohibit` の値をそのまま禁止事項として列挙する。
`cards_and_icons.icon_limit` の値をそのまま展開する。
`layout_principles.avoid` の値をそのまま禁止表現として列挙する。
`layout_principles.recommend` の値をそのまま推奨表現として列挙する。
`quality_bar.target_impression` の値をそのまま目標印象として展開する。
`quality_bar.avoid_impression` の値をそのまま禁止印象として展開する。
`quality_bar.positioning` の値をそのまま展開する。

---

#### 【プロンプトテンプレート（サムネイル）】

以下のプレースホルダーを04_image_style.txtの対応フィールドの値で埋めてプロンプトを生成する。

```
画像内に『[メインコピー]』という日本語テキストを左上起点で大きく表示。[typography_for_thumbnail.main_title_layout の値]。サブコピー「[サブコピー]」をその下に[typography_for_thumbnail.subcopy_rule の値]で配置する。

[mobile_x_thumbnail_priority.viewing_context の値]。[mobile_x_thumbnail_priority.success_condition の値]。

[layout_principles.structural_anchor の値]。背景は[background_colors.surface の値]と[background_colors.background の値]を基調とする。

左側には[Before状態のビジュアルを1つだけ]を置く。右側には[After状態のビジュアルを1つだけ]を置く。主要要素は[information_density.max_major_elements の値]個以内に絞る。[cards_and_icons.icon_limit の値]。

[information_density.prohibit の値]を禁止する。[layout_principles.avoid の値]を避ける。[layout_principles.recommend の値]を実現する。

[typography.font_family.japanese の値]・Boldで[typography_for_thumbnail.main_title_position の値]の階層構造。

[quality_bar.positioning の値]。[quality_bar.target_impression の値]な印象にする。[quality_bar.avoid_impression の値]な表現は禁止する。
```

## X投稿生成

ユーザーから「X投稿を生成して」と言われた場合、またはステップ8.5として自動実行する場合、以下の手順を実行すること。

### 前提

- 入力：直近の `~/test/note_automation/output/YYYY-MM-DD/article_*.md`（最新の記事ファイル）
- 出力：`~/test/note_automation/output/YYYY-MM-DD/x_posts.md`

### 生成内容

#### ① X Articles（X内完全完結の長文記事）

note記事と同じ内容を、X Articles機能でそのまま投稿する。

**【方針】**
- note記事と同じ内容・同じ情報量で投稿する（抽出・要約・変換は不要）
- X内で価値が完全に完結する。noteへの誘導はしない
- フォロワーとの対話を生む投稿にする
- H2見出しには章番号を付ける（例：`## 1. 見出し名`、`## 2. 見出し名`）

**【末尾ルール】**
- 必ず問いかけを入れる（対話誘発フック）
  - 「あなたはどうしてますか？」
  - 「同じ経験した人いますか？」
  - 「〇〇で困ったことありますか？」
- noteリンクは入れない

#### ② ショートポスト（3パターン）

以下の3つの型で、それぞれ140〜280字のショートポストを生成する。
**いずれの型も末尾に問いかけを入れること（リプライを誘発するため）。**

```
型1（共感・あるある型）:
〇〇、最初マジで意味わからなかった。
「なんでこうなってるんだ…」って〇分悩んだ結果、
原因は〇〇だった。
同じ経験した人いる？

型2（ノウハウ完結型）:
〇〇のコツ、3つだけ。

① 〇〇
② 〇〇
③ 〇〇

これだけで〇〇が変わった。
試した人いたら感想聞かせて。

型3（議論・問いかけ型）:
〇〇ってみんなどうしてる？
自分は〇〇という方法でやってるけど、
もっといいやり方があれば知りたい。
```

### 出力フォーマット

`~/test/note_automation/output/YYYY-MM-DD/x_posts.md` に以下の形式で保存する：

````markdown
# X投稿セット：[記事タイトル]

生成日：YYYY-MM-DD

---

## ① X Articles（そのままX Articles機能に貼る）

（本文）

---

## ② ショートポスト（3パターンから1つ選ぶ）

### 型1（共感・あるある型）
（本文）

### 型2（ノウハウ完結型）
（本文）

### 型3（議論・問いかけ型）
（本文）
````

### 完了報告

保存後、以下を報告する：

```
✅ X投稿を生成しました。

📄 保存先：~/test/note_automation/output/YYYY-MM-DD/x_posts.md

運用手順：
1. ① X Articlesの本文をX Articles機能に貼って投稿する
2. ② ショートポストは別の日に単独で投稿できます（連投不要）
```

## GitHubへのpush手順

ユーザーから「pushして」と言われたら、以下を実行すること。

```bash
git -C ~/test add note_automation/input/ note_automation/output/
git -C ~/test commit -m "記事と画像を更新"
git -C ~/test push origin main
```

- 新規ファイルも既存ファイルの変更も、両フォルダ内のものはすべて対象になる
- `.env`（APIキー）は`.gitignore`で除外済みなので誤ってpushされる心配はない
