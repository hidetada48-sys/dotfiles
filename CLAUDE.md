# 個人ルール

## 基本
- 常に日本語で回答する
- 説明は素人にもわかるようにシンプルに
- 専門用語を使う場合は必ず意味を添える

## ★回答提示の絶対ルール（HTML＋リンク付きURL）★最優先・例外なし
- **すべての指示への回答は、原則 HTML にして提示する**。ユーザーが「チャットで答えろ」等と明示的に指示したときだけ、チャット本文に直接書く。明示がなければHTML化が既定。
- **提示は必ず「リンク付きURL」で行う**＝ `[題名](URL)` のクリックできる markdown リンク形式。**生URLの平文提示・チャットへの中身の貼り付けは禁止**（何度も叱責された最頻ミス）。
- **Windows・Linux 両方でクリックできる markdown リンク形式（`[題名](URL)`）に統一する**。OSで出し方を分けない（旧「Linuxは xdg-open で開く」運用は本ルールに置き換え）。
- 判断を挟むと落ちるので、HTML化ツール（プロジェクトの `report_html.py` 等）で変換し、ローカル配信サーバのURLを `[題名](URL)` で返す、を機械的に守る。
- 詳細な失敗例と対策は past_mistakes M-013 を参照。

## コードについて
- コードには必ず日本語コメントを入れる
- 変更前に何をするか説明し、承認を得てから実行する
- 大きな変更の前は必ず確認を取る
- アプローチに自信がなければ、実施前にその旨を伝える

## Git
- コミットメッセージは日本語で書く
- 作業前にブランチを確認する
- git pull前にgit statusを確認する
- 未コミットの変更があれば内容をユーザーに見せて「破棄するか保持するか」確認してからpullする

## 保存3点セット ★区切りでは必ず揃える
作業の区切り（一段落・成果物ができた・「今日はここまで」等）では **①コミット ②プッシュ ③記憶** を必ず1組で提案・実行する。
- ①だけ／①②だけで止めない。Claudeから3点セットを揃えて提案する。
- ③記憶 ＝ 使っている記憶の仕組みへの保存（basic-memory への write_note ＋必要に応じ auto-memory）。Gitに残しても③を飛ばさない。
- ユーザーが①〜③のいずれか（「コミットして」「記憶して」「プッシュして」）を指示したら、残り2点の実行漏れを確認し、まとめて提案する。

## 回答スタイル
- ステップバイステップで説明する
- 一度に大量の情報を出さない
- 不明点は作業前に確認する
- コンテキストがいっぱいになりそうになったら、/compactコマンドを提案する

## 固まり（フリーズ）防止 ★最優先で守る
固まりは2種類。どちらも「一度に大量を出さない」で防ぐ。
- ① ツール呼び出しparse失敗（自分で制御できる）
  - Bashは1コマンド1目的。複雑な正規表現・パイプ・複数リダイレクトを1行に詰めない
  - 出力を絞る（全文ダンプしない）
  - Edit/Write も同じ。長文・表を一度に詰めず、小さく刻む
- ② 長文生成が遅いだけ（サーバー側・制御不能）
  - 唯一の対策は出力を短くする

## Claude Code機能の実装ルール
- 新機能・設定変更を実装する前に必ず公式ドキュメントを確認する
- まとめ記事やツイートの情報だけで実装しない
- 複雑なタスクの前は `/effort high` を実行する（effortLevelのバグ対策）
- 新機能・スクリプトを作る前は必ず brainstorming スキルを発動する

## 作業完了の定義
- 動かせるものは実際に動かして結果を確認してから完了とする
- 「たぶん動くはず」で終わらせない
- エラーが出たら原因を特定してから次に進む

## マルチ環境対応ルール
- 設定・スクリプト・フックを追加・変更したら、必ず以下を確認する
  1. ファイルが `~/dotfiles/` 配下に置かれているか（git管理対象か）
  2. `~/.claude/` 配下の対応ディレクトリがシンボリックリンクになっているか
  3. パスが絶対パス（`/home/hidetada48/...`）ではなくチルダ形式（`~/.claude/...`）か
- Windows PC（miryo）でも動作するよう、ツール・コマンドが未インストールの場合でもスキップできる設計にする
- 作業完了時は「Windows PCでも有効か」を確認項目に含める
- git管理下の共有ファイルを変更する前に「もう一方のOSでも動くか」を確認する
  対象：settings.json / setup.sh / scripts/ / CLAUDE.md / githooks/

## ウェブ検索ルール
- ウェブ検索が必要なときは必ず `mcp__jina__search_web` を使う（WebSearchは使わない）
- URLの内容を読むときは `mcp__jina__read_url` を使う（WebFetchは使わない）
- Jinaが使えない場合（セッション開始直後など）はその旨をユーザーに伝えてから代替手段を確認する

## 記憶の参照ルール
- 過去の作業・記憶・経緯について聞かれたら、必ず以下の両方を確認してから回答する
  1. auto-memory: ~/.claude/projects/【現在のプロジェクトパス】/memory/MEMORY.md とその参照ファイル
  2. basic-memory: mcp__basic-memory__recent_activity および関連ノートを mcp__basic-memory__read_note で読む
- キーワードの有無に関わらず、記憶に関わる質問には必ずこの手順を踏む

## 記憶の扱いルール
- 残課題・未対応（❌）が記憶に記録されていても、そのまま報告しない
  必ず git log・ファイル・コードで現在の実態を確認してから結論を出す
- タスク完了時は関連する auto-memory / basic-memory の未対応表記をその場で更新する
  「作業したが記憶を直さなかった」は作業未完了と同じ
- 作業中に記憶と実態の乖離を発見したら、ユーザーの指示を待たず即座に修正する
