# 個人ルール

## 基本
- 常に日本語で回答する
- 説明は素人にもわかるようにシンプルに
- 専門用語を使う場合は必ず意味を添える

## コードについて
- コードには必ず日本語コメントを入れる
- 変更前に何をするか説明してから実行する
- 大きな変更の前は必ず確認を取る

## Git
- コミットメッセージは日本語で書く
- 作業前にブランチを確認する
- git pull前にgit statusを確認する
- 未コミットの変更があれば内容をユーザーに見せて「破棄するか保持するか」確認してからpullする

## 回答スタイル
- ステップバイステップで説明する
- 一度に大量の情報を出さない
- 不明点は作業前に確認する
- コンテキストがいっぱいになりそうになったら、/compactコマンドを提案する

## Claude Code機能の実装ルール
- 新機能・設定変更を実装する前に必ず公式ドキュメントを確認する
- まとめ記事やツイートの情報だけで実装しない
- 複雑なタスクの前は `/effort high` を実行する（effortLevelのバグ対策）

## 作業完了の定義
- 動かせるものは実際に動かして結果を確認してから完了とする
- 「たぶん動くはず」で終わらせない
- エラーが出たら原因を特定してから次に進む

## マルチ環境対応ルール
- 設定・スクリプト・フックを追加・変更したら、必ず以下を確認する
  1. ファイルが `~/dotfiles/` 配下に置かれているか（git管理対象か）
  2. `~/.claude/` 配下の対応ディレクトリがシンボリックリンクになっているか
  3. パスが絶対パス（`/home/hidetada48/...`）ではなくチルダ形式（`~/.claude/...`）か
- Windows PC（miryo）でも動作するよう、ツール・コマンドが未インストールの場合でもスキップできる設計にする
- 作業完了時は「Windows PCでも有効か」を確認項目に含める
- git管理下の共有ファイルを変更する前に「もう一方のOSでも動くか」を確認する
  対象：settings.json / setup.sh / scripts/ / CLAUDE.md / githooks/

## 記憶の参照ルール
- 過去の作業・記憶・経緯について聞かれたら、必ず以下の両方を確認してから回答する
  1. auto-memory: ~/.claude/projects/【現在のプロジェクトパス】/memory/MEMORY.md とその参照ファイル
  2. basic-memory: mcp__basic-memory__recent_activity および関連ノートを mcp__basic-memory__read_note で読む
- キーワードの有無に関わらず、記憶に関わる質問には必ずこの手順を踏む

## 記憶の扱いルール
- 残課題・未対応（❌）が記憶に記録されていても、そのまま報告しない
  必ず git log・ファイル・コードで現在の実態を確認してから結論を出す
- タスク完了時は関連する auto-memory / basic-memory の未対応表記をその場で更新する
  「作業したが記憶を直さなかった」は作業未完了と同じ
- 作業中に記憶と実態の乖離を発見したら、ユーザーの指示を待たず即座に修正する
